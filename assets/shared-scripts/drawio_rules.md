# DrawIO 绘图标准

Meta-model-agent 产出 DrawIO 技术路线图/过程图/架构图时参考此文件。

## 工具链

- `draw.io.exe`（已打包在 `runtime/draw.io/`，PATH 已注入）

- 导出：`draw.io.exe --export --format pdf --crop --output 图表/xxx.pdf 图表/xxx.drawio`

- ⛔ 务必加 `--crop` 裁剪白边

## ⛔ 零容忍准则（CLI headless 导出踩坑总结）

1. **禁止 `shadow=1`**：CLI 导出时 shadow 导致节点不渲染

2. **禁止 XML 注释 `<!-- -->`**：浪费 token，可能导致解析错误

3. **禁止特殊形状**：`shape=callout` 等在 CLI 导出不稳定，全部用标准 `rounded=1` 圆角矩形。例外：`shape=mxgraph.arrows2.arrow` 可用于过渡箭头（已校验 CLI 导出正常）

4. **换行用 HTML**：`html=1` 模式下用 `&lt;br&gt;` 换行，全部节点务必有 `html=1`。⛔ 涵盖 edge 上的标签文字节点（如判定分支的"是/否"标签）——若标签的 value 里涵盖 `&lt;font&gt;` 等 HTML 标签但 style 里没有 `html=1`，导出后会呈现原始 HTML 代码而不是渲染后的文字

5. **ID 全局唯一**，不可重复

6. **XML 转义**：`<>&"` → `&lt;&gt;&amp;&quot;`

7. **各个 edge 务必有子元素**：`<mxGeometry relative="1" as="geometry"/>` 不可自闭合

8. **中文直接写 UTF-8**

9. **页面背景透明**：`mxGraphModel` 加 `background=none`，并设 `page="0"` 去掉页面边框，避免导出 PDF 有灰色背景

10. **架构图/变量关系图用 TikZ 而非 DrawIO**：CLI 导出不支撑 ELK 自动化路由，`orthogonalEdgeStyle` 的连线会穿过中间节点。需精确连线路由、数学公式、几何图形的图继续用 TikZ

11. **图内不写标题**：标题由 LaTeX `\caption{}` 统一管理，DrawIO 图内不要放标题文字节点（避免标题重复、字体不一致）

## ⛔ 通用设计标准（全部 DrawIO 图务必遵守）

以下标准适用于全部类别的 DrawIO 图（技术路线图/过程图/框架图/指标体系图等）。

### 容器与层次

- **用 swimlane 容器表示阶段/分组**：各个逻辑阶段用 `swimlane` 容器包裹，有标题栏+淡色背景。子节点用 `parent="容器id"` + 相对坐标

- **不同工作区用灰阶层次区分**：容器背景在白色、浅灰和极浅灰之间交替；主要层级依靠边框粗细、虚实和留白表达，不依赖多彩填充

- **容器标题栏**：`startSize=22`，字号 11px，加粗，写明"第N阶段：阶段名称"

### 节点设计

- **节点内容要丰富**：主标题（加粗 9-10px）+ 灰色副标题（8px，`color:#666`），用 `<br>` 换行。避免只写一个词

- **关键节点突出**：用加粗黑灰边框（`strokeWidth=2.2`）+ 极浅灰填充（如 `fillColor=#F2F2F2`）+ ★ 前缀标记

- **结论/结果节点**：用浅灰填充、双线或加粗边框配合 ✓ 标记；仅在确有风险语义时使用深灰虚线，不以红绿作为主要区分

- **负面/异常节点**：用深灰虚线边框（`strokeColor=#555555;dashed=1`）标注，如"虹吸效应""普通面板回归"

- **普通节点**：白底 + 中灰边框（`fillColor=#FFFFFF;strokeColor=#666666`）

### 连线标准

- **用 source/target 属性连线**：`edge="1" source="n1" target="n2"`，避免用人工坐标点。DrawIO 会自动化计算连接位置

- **同容器内连线**：`parent="容器id"`，统一使用深灰或黑色细线

- **跨容器过渡箭头**：`parent="1"`，用粗灰色（`strokeWidth=4;strokeColor=#C0C0C0;endArrow=block;endFill=1;endSize=8`）

- **决策分支**：是/否通过实线与虚线、标签或箭头样式区分，标签紧贴箭头，避免依赖颜色

### ⛔ 连线路由准则（7 条，防止连线穿过节点）

1. **禁止多条边共享同一路线**：若两条边连接同一对节点，务必从不同位置出入。用 `exitY=0.3` 和 `exitY=0.7` 区分，避免都走 0.5

2. **双向连接用对侧**：A→B 从 A 的右侧出（`exitX=1`），进 B 的左侧（`entryX=0`）；B→A 反过来

3. **务必明示指定 exitX/exitY/entryX/entryY**：每条边的 style 里都要写这 4 个属性，避免让 DrawIO 自动化猜

4. **连线务必绕开中间节点**：画边此前先看源和目标之间有没有其他节点挡着，有的话用 waypoints 绕行，留 20-30px 间距

5. **先规划布局再画线**：把节点按流向分层/分列排好，留出连线通道（150-200px 间距），再画边

6. **复杂路由用多个 waypoints**：一个拐点不够就用 2-3 个，形成 L 形或 U 形路线，各个方向变化都需一个 waypoint

7. **连接点选自然方向**：上下流向从底部出（`exitY=1`）顶部进（`entryY=0`）；左右流向从右侧出（`exitX=1`）左侧进（`entryX=0`）。避免用角落连接点（如 `exitX=1,exitY=1`）

**画完后自检**：有没有边穿过非源/目标的节点？有没有两条边重叠？有没有用了角落连接点？

### 整体布局

- **画布宽度 920-950px**：内容区充足宽，节点不挤

- **容器间距 15-17px**：过渡箭头占据这个间距

- **节点宽度 130-220px**：依据文字长度调整，中文节点 ≥ 字数×14+40px

- **节点高度 28-48px**：单行 28px，双行（主标题+副标题）40-48px

- **同行节点等距分布**：计算容器宽度，均匀分配

- **⛔ 节点务必居中分布**：若一行只有 3 个节点但容器宽度能放 4 个，节点要居中排列，避免左对齐留大片空白。计算方法：`左边距 = (容器宽度 - 节点总宽度 - 节点间距总和) / 2`。最后一行（结论/交付）尤其容易出现左对齐问题，务必核验

### 配色方案（swimlane 容器）——黑白学术风格

⛔ 默认使用黑、白与灰阶，确保打印、复印和灰度显示时仍能辨认。可保留一个低饱和强调色，但只能用于极少数关键节点，不能按工作节点逐色装饰。

| 阶段序号 | 容器填充 | 容器边框 | 标签填充 | 节点边框 | 适合内容 |

|---------|---------|---------|---------|---------|---------|

| 1 | `#FFFFFF` | `#555555` | `#333333` | `#555555` | 背景/问题/输入 |

| 2 | `#F7F7F7` | `#777777` | `#444444` | `#666666` | 数据/指标/构建 |

| 3 | `#EEEEEE` | `#666666` | `#333333` | `#555555` | 分析/检验/探索 |

| 4 | `#FFFFFF` | `#444444` | `#222222` | `#444444` | 关键模型/建模 |

| 5 | `#F5F5F5` | `#777777` | `#444444` | `#666666` | 稳健性/检验 |

| 6 | `#EAEAEA` | `#555555` | `#222222` | `#444444` | 结论/推荐 |

节点配色原则：

- 普通节点：白底（`#FFFFFF`）+ 中灰边框，边框宽度 1.3-1.5px

- 关键/高亮节点：极浅灰填充（`#F2F2F2`）+ 深灰边框 2.2px

- 结论/通过节点：浅灰填充（`#E8E8E8`）+ 黑灰双线或加粗边框

- 负面/失败节点：白底 + 深灰虚线边框

- 连线颜色：统一使用 `#555555` 或 `#333333`，宽度 1.3-1.5px

- 过渡箭头：中灰（`#B8B8B8`），宽度 4px

⛔ 节点数量控制：各个阶段/容器内最多 2 行节点，每行最多 4 个。总节点数不超过 30 个。宁可精简也避免塞太多——杂乱比简洁更差。

## 实践核验清单

产出每张 .drawio 后务必逐条自检：

- [ ] **字号核验**：最小字号 ≥ 10px，普通节点 ≥ 11px，标题 ≥ 12px

- [ ] **重叠核验**：相邻节点边界框不交叉，间距 ≥ 8px

- [ ] **对齐核验**：同行节点 y 坐标保持一致，同列节点 x 坐标保持一致

- [ ] **连线遮挡核验**：无连线穿过节点或文字

- [ ] **文字溢出核验**：节点完全涵盖文字，中文节点 width ≥ 字数×14+30px，英文节点 width ≥ 字符数×8+30px。⛔ 若节点内有换行（`<br>`），height 需额外加 16px/行

- [ ] **样式一致性**：同类节点 fillColor/strokeColor/fontSize 统一

- [ ] **容器核验**：子节点 parent 指向无误的容器 id，坐标是相对坐标

- [ ] **edge 核验**：各个 edge 有 `<mxGeometry relative="1" as="geometry"/>`

- [ ] **无 shadow**：确认没有 `shadow=1`

- [ ] **无 XML 注释**：确认没有 `<!-- -->`

- [ ] **html=1 核验**：全部 mxCell（涵盖 edge 标签节点）的 style 里都有 `html=1`。若 value 涵盖 `&lt;font&gt;`/`&lt;b&gt;`/`&lt;br&gt;` 但没有 `html=1`，导出后会呈现原始 HTML 代码

- [ ] **紧凑度**：阶段间过渡箭头间距 ≤ 15px，整图避免拉得太长

- [ ] **填充度**：子分组框/节点务必填满层宽度，不留大片空白。两个分组框之间间距 15px

- [ ] **居中核验**：每行节点数少于容器能容纳的最大数量时，节点务必居中分布，避免左对齐。尤其核验最后一行（结论/交付阶段）

- [ ] **尺寸控制**：DrawIO 图不需限制高度，LaTeX 插入时用 `keepaspectratio` + `height` 双约束自动化处理。专注于内容完整、节点不遮挡即可

## 技术路线图 vs 求解过程图（⛔ 务必区分）

| | 技术路线图 | 求解过程图 |

|---|---|---|

| 视角 | 全局——整篇论文从问题到结论 | 局部——单个子问题从输入到交付 |

| 内容 | 多个阶段，每阶段多个节点，展示整体求解思路 | 一个问题的具体环节、判定分支、数据流向 |

| 放置 | 问题重述章节末尾 | 各子问题章节开头 |

| 数量 | 整篇论文 1 张 | 各个子问题 1 张 |

**布局不限制**——Meta-model-agent 可依据具体赛题内容自由发挥布局和视觉设计（纵向/横向/混合/分区/嵌套都可），只要遵守：

1. 零容忍准则（无 shadow、无 XML 注释、无特殊形状等）

2. 通用设计标准（swimlane 容器、source/target 连线、双行节点、阶段配色）

3. 连线路由准则（不穿过节点、明示指定 exitX/exitY）

4. 图内不写标题（由 LaTeX caption 管理）

⛔ 唯一的硬限制：技术路线图和求解过程图不可长得一样——技术路线图要体现"全局多阶段"的视觉层次，求解过程图要体现"单问题环节链"的逻辑流向。

### ⛔ 求解过程图最低要求（不可只画一条直线）

求解过程图务必体现问题求解的逻辑复杂性，不可只是一串节点从上到下排列。最低要求：

1. **不少于 1 个判定分支**（菱形节点 + 是/否标签）——如"数据是否满足正态性？"、"模型精度是否达标？"、"约束是否满足？"

2. **不少于 1 处并行/分叉**——如数据预处理和特征工程与此同时开展，或两种方法对比后选优

3. **节点双行**——主标题（加粗）+ 灰色副标题（具体方法/参数阐明）

4. **循环反馈**（若有）——参数不满足→调整→重新求解的回路箭头

5. **颜色区分环节类别**——输入（蓝色）、处理（绿色）、判定（黄色）、检验（紫色）、交付（红色）

参考示例布局：主过程纵向排列，遇到判定分支时左右分叉（是→继续主过程，否→右侧替代方案或调整参数后回路），颜色按环节类别区分（输入蓝/处理绿/判定黄/检验紫/交付红）

**⛔ 求解过程图绝对避免画右侧工具/方法注释栏**（"工具与方法"那种独立侧栏是技术路线图专属）。求解过程图应简洁，只展示求解环节和逻辑分支，**禁止**在右侧放 pandas/matplotlib/SciPy/numpy 等工具标签。

**⛔ 也避免把工具/库名（如 SciPy minimize、Python DEAP、scikit-learn 等）写进节点副标题。** 副标题只描述**环节本身做什么**（输入/交付/方法的物理或数学含义），不写技术实现。例子：

- ❌ 错：「优化圆弧半径比 / SciPy minimize」

- ❌ 错：「需求热力分析 / scikit-learn KDE」

- ✅ 对：「优化圆弧半径比 / 最短路线目标函数」

- ✅ 对：「需求热力分析 / 核密度估计识别热点」

⛔ **完整 XML 示例**：`工具/example_flow.drawio` 是一个可直接导出的完整求解过程图示例（有分叉、判定、循环反馈）。产出前先 `cat 工具/example_flow.drawio` 参考其 XML 结构和样式。

## 技术路线图产出过程

### 工作节点 1：查阅规划文档并确定内容

```bash

cat 问题分析.md 2>/dev/null || cat 选题规划.md 2>/dev/null || cat 论文规划.md 2>/dev/null

```

### 工作节点 2：确定工作结构

典型科研论文技术路线图 3-5 个阶段：

- 提出问题（背景/假设/数据）

- 分析问题（指标构建/探索性分析/模型选取）

- 解决问题（关键模型/效应分解/稳健性检验）

- 研究成果（结论/政策推荐）

### 工作节点 3：生成 .drawio XML

按三栏布局产出，用 swimlane 容器 + 相对坐标。

### 工作节点 4：导出 PDF

```bash

draw.io.exe --export --format pdf --crop --output 图表/技术路线图.pdf 图表/技术路线图.drawio

```

### 工作节点 5：校验 PDF 非空

```bash

ls -la 图表/技术路线图.pdf

```

若导出失败，核验：shadow=1？XML 注释？特殊形状？ID 重复？标签未闭合？
