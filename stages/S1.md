# S1: Problem Analysis

**Goal**: Transform the problem from "raw text" into an "executable modeling task package", clarifying problem type, sub-questions, variables, constraints, evaluation criteria, literature direction, and technical roadmap.
**Dependency**: S0
**Execution Style**: Temporary sub-agents write directly to their respective output files; the main agent is responsible for aggregation and merging.

---

## Stage Principles

1. First perform problem-level abstraction, then sub-question-level decomposition.
2. Planning group, analysis group, and review group must all be created as temporary sub-agents, destroyed after completion, retaining no state.
3. Sub-agent prompts must be inlined directly in this stage document, no longer relying on external prompt files.
4. Problem analysis must simultaneously output "problem structure + literature clues + variables with units + reviewer-verifiable content."
5. If the problem has multiple valid interpretation paths, retain at least one baseline interpretation and explicitly note uncertainties.

**Fast Mode Streamlined Path (P1-2)**: Skip Step 4 (Reviewer sub-agent); only invoke Planner + Analyst (2 calls). The main agent completes Step 5 aggregation and proceeds directly to the gate.

---

## Execution Protocol

### Step 1: Scan Problem and Inputs
The main agent reads:
- `inputs/problem_text.md`
- All original attachments under `inputs/`
- `CUMCM_Workspace/_skill/references/problem-triage.md`
- `CUMCM_Workspace/_skill/references/problem-taxonomy.md`
- `CUMCM_Workspace/_skill/references/literature-search-guide.md`

The main agent completes:
- Problem type determination (optimization / prediction / evaluation / statistics / graph-network / simulation / hybrid)
- Whether identifiable data files exist
- Candidate sub-question list
- Terms, domain words, and metric words requiring focused verification

---

### Step 2: Invoke Planner Temporary Sub-Agent (Problem Decomposition and Modeling Direction)

**Temporary Sub-Agent Instruction:**

```markdown
You are the Planner temporary analysis agent. Execute the S1 problem analysis task.

## Your Task
You are responsible for reasoning and writing results directly to the specified output file, not for invoking other stages.
Perform structured analysis based on the original problem text and produce results directly usable by subsequent stages.

## File Permissions
- Readable: inputs/*, references/*
- Writable: outputs/problem_analysis/planner_view.md
- Forbidden: all other files

## Required Reading
1. `inputs/problem_text.md`
2. All problem attachments under `inputs/`
3. `CUMCM_Workspace/_skill/references/problem-triage.md`
4. `CUMCM_Workspace/_skill/references/problem-taxonomy.md`
5. `CUMCM_Workspace/_skill/references/literature-search-guide.md`

## Required Outputs
1. Core problem objective: one-sentence summary
2. Problem type judgment: explain why it belongs to this type or is hybrid
3. Sub-question decomposition: list inputs, outputs, constraints, objectives, and dependencies for each
4. Key variable table: variable name / meaning / unit / type / source / pending?
5. Modeling difficulties: at least 3 items
6. Technical roadmap sketch: 3-6 steps described by stage
7. Literature search keywords: one set each in English and Chinese
8. Risk points and uncertainties: at least 3 items
9. Baseline interpretation: the most conservative, easiest-to-implement interpretation
10. Problem restatement: restate the problem using structured items

## Output Requirements
- Use Markdown.
- Language must be concise, directly usable in `analysis.md`.
- No filler phrases; no "consider" statements.
- Each sub-question must have an executable description.
- Recommended length: 400-600 words (information density over word count).
```

The sub-agent writes the result directly to `outputs/problem_analysis/planner_view.md`.

---

### Step 3: Invoke Analyst Temporary Sub-Agent (Data and Usability Assessment)

**Temporary Sub-Agent Instruction:**

```markdown
You are the Analyst temporary data assessment agent. Execute the S1 data availability check.

## Your Task
You are responsible for reasoning and writing results directly to the specified output file, not for invoking other stages.
You must first determine whether the problem depends on data, then decide the output depth.

## File Permissions
- Readable: inputs/*, outputs/problem_analysis/planner_view.md, references/*
- Writable: outputs/problem_analysis/analyst_view.md
- Forbidden: all other files

## Required Reading
1. `inputs/problem_text.md`
2. All data files under `inputs/`
3. `outputs/problem_analysis/planner_view.md`
4. `CUMCM_Workspace/_skill/references/problem-triage.md`
5. `CUMCM_Workspace/_skill/references/problem-taxonomy.md`

## Branch Rules
- If `equation_driven` or explicitly data-free:
  - Briefly explain why data requirements are minimal
  - List potential external data or parameters needed
  - Provide data risk warnings for subsequent stages
- If `data_driven` or hybrid:
  - Output complete data profile
  - Output data field dictionary
  - Output quality issues and cleaning suggestions
  - Output variable-to-data-field mapping

## Required Outputs
1. Data availability judgment
2. Data profile: fields, types, missing values, outliers, range, distribution, correlations
3. Data dictionary: field name / meaning / unit / type / value range / source
4. Data risks: missing values, noise, sample size, timeliness, bias
5. Cleaning suggestions: prioritized
6. Mapping to problem variables
7. Conclusion on whether data is sufficient for subsequent modeling

## Output Requirements
- Use Markdown.
- If no data files exist, still output an explicit "insufficient data" statement.
- Results should be sufficient to support `analysis.md` and subsequent S2.
```

The sub-agent writes the result directly to `outputs/problem_analysis/analyst_view.md`.

---

### Step 4: Invoke Reviewer Temporary Sub-Agent (Review and Rejection)

**Temporary Sub-Agent Instruction:**

```markdown
You are the Reviewer temporary review agent. Execute the S1 quality review.

## Your Task
You are responsible for reviewing and directly writing the report to the specified output file, not for rewriting upstream conclusions.
Your role is to identify gaps, point out logical flaws, and provide a pass/reject verdict.

## File Permissions
- Readable: outputs/problem_analysis/planner_view.md, outputs/problem_analysis/analyst_view.md, references/*
- Writable: outputs/problem_analysis/reviewer_review.md
- Forbidden: all other files

## Review Objects
1. `outputs/problem_analysis/planner_view.md`
2. `outputs/problem_analysis/analyst_view.md`
3. `CUMCM_Workspace/_skill/references/scoring-criteria-cumcm.md`

## Review Dimensions
1. Is the problem understanding accurate?
2. Is the sub-question decomposition complete?
3. Are variables and units consistent?
4. Do data and problem match?
5. Is subsequent modeling executable?

## Output Requirements
1. Review summary: 50-100 words
2. Four- or five-dimension score table: score + explanation per dimension
3. Issue list: sorted by P0 / P1 / P2
4. Fix suggestions: actionable items
5. Verdict: PASS / CONDITIONAL / REJECT

## Decision Criteria
- Missing key variables, incomplete sub-questions, incorrect data judgment: REJECT
- Minor flaws fixable through corrections: CONDITIONAL
- Clear structure, executable, informationally closed: PASS
```

The sub-agent writes the result directly to `outputs/problem_analysis/reviewer_review.md`.

### Step 5: Main Agent Synthesis

The main agent synthesizes the three temporary sub-agent outputs into the following artifacts:

1. `outputs/problem_analysis/analysis.md`
2. `outputs/problem_analysis/symbol_table.md`
3. `outputs/problem_analysis/subproblems.json`
4. `outputs/literature/literature_review.md`
5. `outputs/problem_analysis/problem_classification.json`
6. `outputs/problem_analysis/route_map.md` (if required by project convention)

During synthesis, ensure:
- Remove duplicate expressions
- Unify variable names, units, and symbols
- Clarify input-output relationships for each sub-question
- Clarify dependency order for subsequent stages
- Write an executable modeling path, not just an overview

---

### Step 6: Generate Technical Roadmap

If the problem is suitable for visualization, the main agent generates a technical roadmap and its narrative description as the basis for Figure 1 in the subsequent paper.

The roadmap should cover at least:
- Problem understanding
- Data governance
- Model selection
- Experiment race
- Modeling & solving
- Verification and evidence chain

## Gate G1

### Must Satisfy
- [ ] `analysis.md` exists and contains a structured problem restatement
- [ ] Sub-question decomposition is complete, with input / output / constraints / metrics for each sub-question
- [ ] Symbol table is complete with clear units
- [ ] Literature review includes method comparison or search rationale
- [ ] Data availability judgment is complete
- [ ] Reviewer verdict is not `REJECT`

### Check Command
```bash
python CUMCM_Workspace/_skill/scripts/gate_contracts.py g1
python CUMCM_Workspace/_skill/scripts/stage_executor.py gate_check S1
```

### Gate Failure -> Auto Rework (Hard-Coded, Cannot Be Skipped)

```bash
IF gate_check returns FAIL:
    1. Record failed gate name and failure reason (missing analysis.md / incomplete sub-questions / missing symbol table / no literature review / reviewer REJECT)
    2. Run python CUMCM_Workspace/_skill/scripts/stage_executor.py rework S1 --reason "Gate G1 failed: <specific reason>"
    3. Return to Step 2 (re-invoke Planner sub-agent) or Step 3 (re-invoke Analyst sub-agent) based on failure type
    4. rework count +1
    5. If rework >= 2: log to state/decision_log.json, generate failure report, block and prompt human intervention
    6. If failure reason is "insufficient upstream data": run python CUMCM_Workspace/_skill/scripts/stage_executor.py rework S0 --reason "S1 found insufficient problem registration: <reason>", fallback to S0
```

**This logic cannot be omitted. Gate failure must execute the rework command, record count, and block on exceed.**
