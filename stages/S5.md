# S5: Modeling & Solving -- Per-Subquestion Implementation

**Goal**: Based on the primary model confirmed in S4, complete modeling derivation, code implementation, on-demand figure generation, result interpretation, and structured summary for each sub-question. Form a complete solution pipeline ready for verification, while accumulating sufficient visualization material for S8 paper writing.
**Dependency**: S4
**Execution Style**: Temporary sub-agents write directly to their respective output files (isolated by sub-question: `outputs/q{N}/`); the main agent handles cross-subquestion merging and unified standards.

---

## Stage Principles

1. All sub-agents are temporary calls that **write directly to their respective output files**.
2. The main agent is responsible for cross-subquestion merging, unified naming, and unified variable standards.
3. Each sub-question must first establish a baseline, then build the primary model, then perform quantitative comparison -- Baseline-First is not optional.
4. Neither code-only nor conclusion-only output is permitted; derivation, implementation, and explanation must all be provided.
5. Figure generation is driven by need: no preset figure types. Based on the sub-question's data characteristics, model properties, and paper narrative requirements, the Builder sub-agent determines which figures to generate, of what type, and in what style.
6. All figures must meet publication quality: clear axis labels, legends, titles, and color schemes -- directly embeddable in the paper body.
7. Proposer and Builder sub-agents analyze based on existing literature in the `CUMCM_Workspace/_skill/references/` directory. If supplementary literature is needed, annotate `[Suggest supplement: keywords xxx]`; the main agent performs web_search uniformly.

---

### Step 2: Loop Steps 3-7 for Each Sub-Question Q{N}

Execute for `Q1, Q2, ..., Qn` one by one. Each sub-question must complete:
1. Proposer derivation -> 2. Builder implementation + figures -> 3. Run validation -> 4. Structured summary
No step may be skipped for any sub-question.

### Step 3: Invoke Proposer Sub-Agent -- Model Derivation

**Temporary Sub-Agent Instruction:**

```markdown
You are the Proposer temporary modeling derivation agent, existing only in this round. Complete the primary model derivation for sub-question Q{N}.

## Role Constraints
- You are responsible for derivation and writing results directly to the specified output file.
- You must provide a simple baseline first, then the primary model.
- If the primary model is significantly more complex than the baseline, you must justify the complexity.

## Literature Analysis Protocol
- Analyze modeling methods based on existing literature in the `CUMCM_Workspace/_skill/references/` directory
- Search for methodologies in `CUMCM_Workspace/_skill/references/model-formulation-guide.md` and `CUMCM_Workspace/_skill/references/algorithm-library/`
- If supplementary literature is needed, annotate: `[Suggest supplement: keywords xxx]` (main agent searches uniformly)

## Required Inputs
1. `methods/model_selection_report.md`
2. `outputs/problem_analysis/subproblems.json`
3. `outputs/problem_analysis/symbol_table.md`
4. `data/data_dictionary.md`
5. `data/data_profile_report.md`
6. References: `CUMCM_Workspace/_skill/references/model-formulation-guide.md`, `CUMCM_Workspace/_skill/references/algorithm-library/`

## Task Objectives
1. Clarify the sub-question's mathematical objective, variables, constraints, and assumptions.
2. First establish an interpretable simple baseline (mean/median/linear approximation/greedy/random -- choose the appropriate one).
3. Build the primary model on top of the baseline and explain the choice.
4. Define quantitative comparison metrics between the primary model and baseline.
5. Produce a derivation structure suitable for paper writing (formula numbering, unified symbols).

## Output Requirements
Output in Markdown:
- Sub-question definition and mathematical formulation
- Variables and symbols (aligned with symbol_table.md)
- Assumptions and constraints
- Baseline model (formula + explanation)
- Primary model (formula + derivation + explanation)
- Evaluation metric definitions
- Result interpretation framework
- Input/output conventions for Builder
- Literature reference annotations (if supplementary needed: `[Suggest supplement: keywords xxx]`)
```

The sub-agent writes the derivation result directly to `methods/solving/Q{N}/proposer_derivation.md`.

The main agent verifies the file exists and is non-empty, and checks derivation completeness (formulas, symbols, baseline, evaluation metrics). If anything is missing, supplement and continue.

### Step 4: Invoke Builder Sub-Agent -- Code Implementation + On-Demand Figure Generation

**Temporary Sub-Agent Instruction:**

```markdown
You are the Builder temporary implementation agent, existing only in this round. Implement runnable code for sub-question Q{N}, generate on-demand figures, and organize result output.

## Role Constraints
- You are responsible for reasoning, implementation suggestions, and result analysis; write code and results directly to the specified output file.
- Code must fix random seeds and specify input/output paths clearly.

## File Permissions
- Readable: Proposer derivation results, methods/model_selection_report.md, data/*, references/*
- Writable: src/q{N}/, outputs/q{N}/ (all output files)
- Forbidden: other sub-questions' files, state/*
- You must implement the baseline first, then the primary model.

## Literature Analysis Protocol
- Analyze implementation approaches based on existing literature in the `CUMCM_Workspace/_skill/references/` directory
- Search for code references in `CUMCM_Workspace/_skill/references/algorithm-library/`
- If supplementary literature is needed, annotate: `[Suggest supplement: keywords xxx]` (main agent searches uniformly)

## Required Inputs
1. Proposer derivation results (provided by main agent after organization)
2. `methods/model_selection_report.md`
3. `outputs/problem_analysis/symbol_table.md`
4. Available data under `data/processed/`
5. References: `CUMCM_Workspace/_skill/references/algorithm-library/`, `CUMCM_Workspace/_skill/references/robustness-guide.md`

## Task Objectives

### A. Code Implementation
1. Output runnable Python code structure (baseline + primary model).
2. Code must include: input validation, error handling, log output, result saving.
3. Fix random seeds for reproducibility.

### B. On-Demand Figure Generation (Core Requirement)
**Principle: No preset figure types; decide based on actual needs.**

You must determine which figures to generate based on the following dimensions:

1. **Data Exploration Dimension**:
   - Data distribution characteristics -> histogram/boxplot/violin plot/KDE (choose as needed)
   - Variable relationships -> scatter plot/heatmap/parallel coordinates (choose as needed)
   - Time series characteristics -> line chart/area chart/seasonal decomposition (choose as needed)

2. **Model Results Dimension**:
   - Model performance comparison -> bar chart/radar chart/pairwise comparison (choose as needed)
   - Predicted vs actual -> scatter + diagonal/residual plot/Q-Q plot (choose as needed)
   - Convergence process -> loss curve/fitness curve (choose as needed)

3. **Paper Narrative Dimension**:
   - Methodology showcase -> flowchart/architecture diagram/algorithm pseudocode box (choose as needed)
   - Result highlights -> the figure type that best showcases the core finding (free judgment)
   - Comparative argument -> the most intuitive display of baseline vs primary model (free judgment)

4. **Problem-Specific Dimension**:
   - Optimization problems -> feasible region visualization/Pareto front/objective contour (choose as needed)
   - Classification problems -> confusion matrix/ROC curve/decision boundary (choose as needed)
   - Regression problems -> fitted curve/residual analysis/prediction interval (choose as needed)
   - Clustering problems -> silhouette plot/dendrogram/dimension-reduced scatter (choose as needed)

**Figure Quality Requirements (Publication-Grade)**:
- All figures must have: axis labels, legends, titles, appropriate color schemes
- Font size suitable for paper embedding (no smaller than 10pt)
- Resolution >= 300 DPI
- Each figure must include a 50+ word caption explaining what it shows and the core finding
- Figure naming convention: `fig_{subquestion}_{descriptive_name}.png`

**Minimum Quantity Requirements**:
- Standard mode: at least 3 figures per sub-question
- Championship mode: at least 5 figures per sub-question
- At least 1 "paper highlight figure" (the figure that best represents the sub-question's core finding)

### C. Result Output
1. Run code and produce result tables.
2. Output `summary.md` structured summary.
3. Output `baseline_comparison.md`.

## Output Requirements
Output in Markdown:
- Code design description (with literature reference annotations)
- Input/output definitions
- Execution steps
- **Figure generation plan** (list each figure: type, purpose, filename, intended message)
- Result tables
- Baseline comparison
- Constraint self-check
- File list for main agent to persist (including all figures)
```

The sub-agent writes code, figures, and results directly to `src/q{N}/` and `outputs/q{N}/`.

The main agent verifies that the following files exist and are non-empty: `src/q{N}/*.py`, `outputs/q{N}/tables/*.csv`, `outputs/q{N}/figures/`, `outputs/q{N}/summary.md`.

---

### Step 5: Main Agent Runs Code and Validates

The main agent executes:
1. Run baseline code, record baseline metrics
2. Run primary model code, record primary model metrics
3. Verify numerical reasonableness (no NaN/Inf/negative value anomalies)
4. Compare quantitative metrics between baseline and primary model
5. If the primary model improvement is less than 5%, explain the necessity of the complex model in summary.md

---

### Step 6: Main Agent Writes Sub-Question Artifacts

Confirm all the following files exist and are non-empty:
- `src/q{N}/*.py` -- runnable code
- `src/q{N}/baseline.py` -- baseline code
- `src/q{N}/baseline_comparison.md` -- baseline comparison
- `outputs/q{N}/tables/*.csv` -- result tables
- `outputs/q{N}/figures/*.png` -- on-demand generated figures
- `outputs/q{N}/figure_captions.md` -- figure caption summary
- `outputs/q{N}/summary.md` -- structured summary

`summary.md` must include:
- Key formulas and derivation
- Algorithm flow
- Result tables and figure interpretations (referencing each figure)
- Constraint validation statement
- Domain meaning explanation
- Baseline comparison conclusion

---

### Step 7: Cross-Subquestion Merge and Consistency Check

After all sub-questions are complete, the main agent executes:
1. Symbol conflict detection -- does the same symbol name have different meanings across sub-questions?
2. Unit consistency -- is the same physical quantity in consistent units across sub-questions?
3. Output metric unification -- is the calculation method of evaluation metrics consistent?
4. Figure style unification -- are color schemes, fonts, and figure styles consistent across sub-questions?
5. Write `outputs/unified_framework.md` (cross-subquestion unified framework)
6. Write `outputs/cross_validation_report.md` (cross-validation report)
7. Update `outputs/problem_analysis/symbol_table.md` (merge all sub-question symbols)

### Step 9: Gate Check

Run parallel merge quality gate:

```

```

Run code review gate:
```bash

```

Run math sandbox gate:
```bash

```

**Gate failure -> auto rework (hard-coded, cannot be skipped)**:
```bash
IF any gate returns FAIL:
    1. Record failed gate name and failure reason
    2. Run python CUMCM_Workspace/_skill/scripts/stage_executor.py rework S5 --reason "<gate_name>: <reason>"
    3. Locate the problematic sub-question based on failure reason, return to Step 5 (code fix) or Step 3 (derivation redo)
    4. rework count +1
    5. If rework >= 2: log to state/decision_log.json, generate failure report, block and prompt human intervention
```
**This logic cannot be omitted or simplified to "fix and re-check". The rework command must be executed, count recorded, and blocked on exceed.**
