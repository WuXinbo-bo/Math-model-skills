# S5.5: Model Evolution -- Evolutionary Innovation Exploration

**Goal**: Systematically explore variations of the primary model produced in S5. Discover superior variants or verify the optimality of the current model. Produce an evolution log and innovation assessment report to support the paper's model innovation argument.
**Dependency**: S5
**Execution Style**: Temporary sub-agents write directly to their respective output files; the main agent handles evolution experiment execution and aggregation.

---

## Stage Principles

1. This stage can be conditionally skipped: if the problem type is `equation-driven` and S4 has already confirmed a unique analytical/semi-analytical primary model, skip this stage (record skip reason in `state/decision_log.json`). **Fast mode skips this stage by default** (regardless of problem type).
2. If the problem type is `data_driven`, `ML`, or has multiple competing modeling directions (Standard/Championship mode), **this stage must be executed**.
3. Evolution is not blind search: it must be based on the baseline vs primary model comparison results from S5, with targeted variation directions.
4. Failed evolution (no superior variant found) is a valid outcome; it must be honestly recorded and explained.
5. Championship mode must produce an innovation justification paragraph (>= 200 words); Standard mode is recommended but not mandatory.
6. Builder and Proposer sub-agents analyze based on existing literature in the `CUMCM_Workspace/_skill/references/` directory. If supplementary literature is needed, annotate `[Suggest supplement: keywords xxx]`; the main agent performs web_search uniformly.

### Step 2: Invoke Builder Sub-Agent -- Evolution Experiment Design

**Temporary Sub-Agent Instruction:**

```markdown
You are the Builder temporary evolution experiment agent, existing only in this round. Design a systematic model evolution plan for the S5.5 stage.

## Role Constraints
- You are responsible for reasoning and experiment design; write results directly to the specified output file.
- Evolution must be based on the primary model confirmed in S5, not starting from scratch.

## File Permissions
- Readable: outputs/q{N}/summary.md, methods/model_selection_report.md, src/q{N}/*, references/*
- Writable: outputs/evolution/Q{N}/evolution_plan.md
- Forbidden: all other files
- Each evolution direction must be justified (S5 weaknesses, alternative methods from literature, performance bottleneck analysis).

## Literature Analysis Protocol
- Analyze evolution methods based on existing literature in the `CUMCM_Workspace/_skill/references/` directory
- Search for evolutionary algorithms in `CUMCM_Workspace/_skill/references/algorithm-library/`
- If supplementary literature is needed, annotate: `[Suggest supplement: keywords xxx]` (main agent searches uniformly)

## Required Inputs
1. S5 sub-question results: `outputs/q{N}/summary.md`
2. `methods/model_selection_report.md`
3. `src/q{N}/*.py` (S5 primary model code)
4. References: `CUMCM_Workspace/_skill/references/model-formulation-guide.md`, `CUMCM_Workspace/_skill/references/algorithm-library/`

## Task Objectives

### A. Variation Search Space Definition
Based on the S5 primary model structure, define 3-5 variation dimensions:
1. **Parameter variation**: hyperparameter tuning range (learning rate, regularization coefficient, number of layers, etc.)
2. **Structural variation**: model architecture fine-tuning (feature combination, loss function, kernel function, etc.)
3. **Method variation**: alternative algorithm direction (if the primary model is method A, search whether method B performs better)
4. **Ensemble variation**: multi-model fusion methods (weighted averaging, Stacking, Blending)

### B. Evolution Plan Design
Design a 3+ generation evolution process:
- Generation 1: random/grid search baseline
- Generation 2: targeted variation based on Generation 1 results
- Generation 3: elite preservation + local search

Each generation must specify:
- Variation strategy
- Selection criteria
- Termination condition
- Expected computational cost

### C. On-Demand Figure Planning
Plan figures to generate during evolution (no preset types, decide based on actual needs):
- Convergence curves (fitness vs iterations)
- Variation space visualization (dimension-reduced scatter/heatmap)
- Candidate model comparison (bar chart/radar chart)
- Best variant vs base model comparison (pairwise plot)

**Figure Quality Requirements**: Same as S5 -- publication-grade, 300 DPI, axis labels, each with 50+ word caption.

## Output Requirements
Output in Markdown:
- Variation search space definition
- Evolution plan (with generations, strategies, termination conditions)
- Figure generation plan
- Expected output file list
- Literature reference annotations (if supplementary needed: `[Suggest supplement: keywords xxx]`)
```

The sub-agent writes the evolution plan directly to `outputs/evolution/Q{N}/evolution_plan.md`.

The main agent evaluates the plan's feasibility and computational cost. If the plan is unreasonable, request the Proposer to supplement suggestions and adjust.

### Step 3: Main Agent Executes Evolution Experiments

The main agent executes evolution per the Builder-designed plan:
1. Run the evolution process for each sub-question one by one
2. Record the best individual, fitness, and variation details for each generation
3. Save intermediate results to `outputs/evolution/Q{N}/generations/`
4. Generate figures needed during evolution (per the figure plan in Step 2)
5. Run innovation gate check (Championship mandatory):

```bash
python CUMCM_Workspace/_skill/scripts/gate_contracts.py innovation --mode <mode>
```

**If failed**: Request the Proposer to supplement innovation justification in Step 4.

**Temporary Sub-Agent Instruction:**

```markdown
You are the Proposer temporary innovation assessment agent, existing only in this round. Assess the innovation of S5.5 evolution results.

## Role Constraints
- You are responsible for assessment and recommendations; write results directly to the specified output file.
- Innovation assessment must be objective: if there are advantages, state them; if not, honestly say so.

## File Permissions
- Readable: outputs/evolution/Q{N}/*, outputs/q{N}/summary.md, references/*
- Writable: outputs/evolution/Q{N}/innovation_assessment.md
- Forbidden: all other files

## Literature Analysis Protocol
- Analyze innovation based on existing literature in the `CUMCM_Workspace/_skill/references/` directory
- Search for methodological references in `CUMCM_Workspace/_skill/references/model-formulation-guide.md`
- If supplementary literature is needed, annotate: `[Suggest supplement: keywords xxx]` (main agent searches uniformly)

## Required Inputs
1. `outputs/evolution/Q{N}/evolution_plan.md`
2. Evolution logs under `outputs/evolution/Q{N}/generations/`
3. S5 `outputs/q{N}/summary.md`
4. References: `CUMCM_Workspace/_skill/references/model-formulation-guide.md`

## Task Objectives

1. **Innovation Assessment** (Championship mandatory >= 200 words):
   - Innovation of the best variant compared to the S5 base model
   - Innovation mechanism explanation (why it is better)
   - Number of multi-method fusion categories (>= 2 categories is acceptable)
   - Whether cross-disciplinary references exist

2. **Honesty Declaration**:
   - If evolution did not find a superior variant, honestly explain why (insufficient search space? model already optimal? data limitations?)
   - Fabrication of non-existent innovations is prohibited

3. **Evolution Contribution Analysis**:
   - Which variation dimensions contributed the most
   - Which variation dimensions had no significant effect
   - Implications for subsequent stages (S6 sensitivity analysis)

## Output Requirements
Output in Markdown:
- Innovation assessment paragraph (Championship >= 200 words)
- Variation contribution analysis
- Honesty declaration
- Innovation recommendations for paper writing
- Literature reference annotations (if supplementary needed: `[Suggest supplement: keywords xxx]`)
```

The sub-agent writes the innovation assessment directly to `outputs/evolution/Q{N}/innovation_assessment.md`.

The main agent verifies the file exists and is non-empty. If the innovation gate still fails, supplement the justification and re-check.
