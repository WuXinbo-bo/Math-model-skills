# S6: Verification and Sensitivity Analysis

**Stage Goal**: Perform perturbation testing, parameter scanning, error analysis, and cross-validation on the S5 primary model results. Confirm that conclusions are stable, boundaries are clear, and constraints hold.

**Dependency**: `S5` (if S5.5 is enabled, S5.5 takes priority as the preceding stage)

**Execution Principles**:
- All sub-agents are temporary calls that **write directly to their respective output files**.
- Verification is not a formality check -- it must actually demonstrate that results can withstand perturbations.
- Every key conclusion must have counterexample awareness, error margins, and alternative validation.

**Fast Mode Streamlined Path (P1-2)**: Execute only Step 1 (Builder sensitivity analysis) + Step 3-4 (parameter scanning + robustness check). Skip Step 2 (Proposer verification confirmation) and Step 6 (cross-validation). The main agent decides independently whether S5 rework is needed.

## Required Actions -- DO NOT skip any step

### Step 3: Parameter Scanning
Perform +/-10% / +/-20% / +/-50% perturbation on key parameters for each sub-question.

### Step 4: Robustness Check
Perturb each sub-question and record whether conclusions are reversed.

### Step 5: Error Analysis
Calculate confidence intervals, error bounds, or numerical deviation upper limits.

### Step 6: Cross-Validation
If possible, validate key conclusions using independent methods and report consistency.

### Step 7: Constraint Self-Check
Check the following:
- Conservation
- Boundary conditions
- Physical limits
- Residual levels

## Gate Failure -> Distinguish Internal Rework vs S5 Fallback (P1-4)

**Decision Criteria** (must classify before deciding rollback target):

| Failure Type | Condition | Rollback Target |
|-------------|-----------|----------------|
| **Insufficient verification method** | Sensitivity range too narrow, insufficient perturbation levels, inappropriate cross-validation method | `rework S6` (internal rework) |
| **Model derivation error** | Core numerical results reverse under +/-20% perturbation, constraint violations, error bounds exceeded | `rework S5` (fallback to modeling) |

**Hard criteria for "need S5 fallback" (auto-triggered, cannot be subjectively skipped)**:
- Any sub-question's core conclusion reverses under +/-20% perturbation
- Any key parameter's sensitivity > 0.8 (highly sensitive) with no reasonable explanation
- Constraint check reveals physical violations (conservation/boundary conditions not satisfied)
- Error bound exceeds acceptable range (relative error > 30%)

**Criteria for "internal S6 rework"**:
- Sensitivity analysis only covers some parameters (missing critical ones)
- Perturbation range is unreasonable (e.g., only +/-5%, not covering +/-20%/+/-50%)
- Cross-validation method is inappropriate (too similar to the primary model method)
- Robustness check does not cover all sub-questions

```bash
# Internal rework (insufficient verification method)
IF failure type is "insufficient verification method":
    1. Record failure reason (missing parameters / insufficient range / inappropriate method)
    2. Run python CUMCM_Workspace/_skill/scripts/stage_executor.py rework S6 --reason "S6 insufficient verification: <reason>"
    3. Return to S6 Step 1 (re-design sensitivity plan, expand range / supplement parameters)
    4. rework count +1 (counted as S6 internal rework)

# Fallback to S5 (model derivation error)
IF failure type is "model derivation error":
    1. Record failure reason (which sub-question, which parameter is unstable, whether conclusion reversed)
    2. Run python CUMCM_Workspace/_skill/scripts/stage_executor.py rework S5 --reason "S6 verification failed: <reason>"
    3. Return to S5 Step 3 (re-derive + implement for the failed sub-question)
    4. After S5 fix is complete, re-enter S6 Step 1
    5. rework count +1 (counted toward S5->S6 cycle limit)

# Exceed limit blocking
IF S6 internal rework >= 2 times or S5->S6 cycle >= 2 times:
    Log to state/decision_log.json, generate failure report, block and prompt human intervention
```

**This logic cannot be omitted or simplified to "recommendation". Gate failure must first classify the failure type, then select the correct rollback target.**
