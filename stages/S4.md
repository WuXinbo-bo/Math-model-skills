# S4: Experiment Race & Early Code Prototype Validation

**Goal**: Write runnable simplified prototype code (<= 200 lines) for the candidate models selected in S3. Run actual evaluations on the same dataset with the same metrics to determine the primary plan and backup plan.
**Dependency**: S3
**Execution Style**: Temporary sub-agents write directly to their output files. The Builder sub-agent writes the design document; the main agent handles code generation, execution validation, and aggregation.

---

## Stage Principles

1. All candidate models must be compared on the same dataset, with the same evaluation metrics and the same random seed strategy.
2. If the problem is `equation-driven` and the analytical solution is sufficiently stable, it can be simplified to minimal validation before proceeding to S5.
3. If S4 gate fails, automatically fall back to S3 (with failure reasons), max 2 cycles.
4. Prototype code must be actually runnable and produce numerical results; pseudocode is not allowed.
5. The main agent must verify the reasonableness of prototype results; do not skip checks and proceed directly.

**Fast Mode Streamlined Path (P1-2)**: Fast mode skips full S4 experiment race by default. The main agent directly uses the S3 `model_selection_report.md` conclusions, writes "S4 skipped (Fast mode: using S3 verdict directly)" in `outputs/experiments/race/Q1/experiment_race_plan.md`, then proceeds to S5. Must still run `stage_executor.py gate_check S4` to confirm the skip is compliant.

## Execution Protocol

### Step 1: Main Agent Feasibility Judgment

The main agent reads:
- `methods/model_selection_report.md`
- `methods/debate_summary.md`
- `outputs/problem_analysis/subproblems.json`
- `outputs/problem_analysis/data_status.md`

Judgment (P2-8 trigger conditions):
- **Simplified path (Step 2)** trigger (any one of):
  1. Only 1 ADVANCE model and no REPAIR in S3 `debate_summary.md`
  2. `equation_driven` + S3 has already identified a unique primary model
  3. Fast mode defaults to simplified path
- **Full race path (Step 3)** trigger:
  - `data_driven` or multiple competing models exist (>= 2 ADVANCE or REPAIR)

### Step 7: Main Agent Verifies Prototype Result Reasonableness

The main agent must verify:
1. **Numerical reasonableness**: Is the prototype output within the expected range?
2. **Baseline comparison**: Is the primary model better than the baseline? If not, explain why the model was still chosen.
3. **Stability**: Are results consistent across multiple runs?
4. **Failure handling**: If prototype fails to run, record the reason and assess impact.

**If prototype validation fails**:
- Prototype code cannot run -> check dependencies/data paths
- Prototype performance far below expectations -> model selection may be wrong, trigger S3 rework
- Prototype results unstable -> increase random seed count or adjust parameters

---

### Step 8: Iteration Fallback Protocol

When S4 gate is not passed:

```bash
python CUMCM_Workspace/_skill/scripts/stage_executor.py gate_check S4
python CUMCM_Workspace/_skill/scripts/stage_executor.py rework S3 --reason "S4 prototype validation failed: <reason>"
```

Information passed during fallback (S3 must read when re-executing):
1. `outputs/experiments/race/Q1/prototype_results.md`
2. `outputs/experiments/race/Q1/risk_notes.md`
3. Specific failure reasons in S3 rework reason

**Failure reason mapping**:
- All prototypes failed to run -> S3 may have selected overly complex models
- Prototype performance lower than baseline -> S3 model selection may be wrong
- Prototype results unstable -> S3 assumptions may be invalid

**Iteration limit**: S3->S4 cycle max 2 times; beyond that, generate manual decision report.
