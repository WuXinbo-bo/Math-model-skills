# S2: Data Governance

**Goal**: Determine the extent to which the problem depends on data; establish a traceable data profile, field dictionary, and cleaning plan to provide trustworthy inputs for subsequent modeling.
**Dependency**: S1
**Execution Style**: Temporary sub-agents write data governance artifacts directly; the main agent handles branch routing and merging.

---

## Stage Principles

1. First determine whether data is actually needed, then decide governance depth.
2. Temporary sub-agents write directly to their respective output files, retaining no state.
3. If no data files exist, still output an "insufficient data" statement and "external data requirements."
4. Data governance must serve subsequent S3 model selection and S5 implementation.
5. All fields, units, missing values, and outliers must be traceable.

**Fast Mode Streamlined Path (P1-2)**: If S1 determined `equation_driven`, use the lightweight path -- skip the Analyst sub-agent; the main agent directly generates `data/data_status.md` (containing only data requirement notes and external parameter sources), then proceeds to the gate.

---

## Execution Protocol

### Step 1: Main Agent Branch Judgment

The main agent reads:
- `outputs/problem_analysis/analysis.md`
- `outputs/problem_analysis/symbol_table.md`
- All data files under `inputs/`

Determine the problem type:
- `equation_driven`: No explicit data files or data only serves as parameter supplementation
- `data_driven`: Core dependencies on tables, time series, images, text, or structured data
- `hybrid`: Both explicit data and theoretical derivation required

If determined as `equation_driven`:
- Skip heavy cleaning
- Record data requirements and external supplementation suggestions
- Directly generate a lightweight data description
- Still proceed to gate check

---

### Step 2: Invoke Analyst Temporary Sub-Agent (Data Profiling and Cleaning)

**Temporary Sub-Agent Instruction:**

```markdown
You are the Analyst temporary data governance agent. Execute the S2 data governance task.

## Your Task
You are responsible for reasoning and writing results directly to the specified output file, not for invoking other stages.
You need to determine whether data is sufficient, whether fields are usable, and whether cleaning is traceable.

## Required Reading
1. `outputs/problem_analysis/analysis.md`
2. `outputs/problem_analysis/symbol_table.md`
3. All data files under `inputs/`
4. `CUMCM_Workspace/_skill/references/problem-triage.md`
5. `CUMCM_Workspace/_skill/references/literature-search-guide.md`

## Required Outputs
1. Data availability conclusion: sufficient / partially sufficient / insufficient
2. Data profile: fields, types, missing rate, outliers, duplicates, distribution, correlations
3. Data dictionary: field name / meaning / unit / type / value range / corresponding variable
4. Data cleaning plan: prioritized, with purpose of each step
5. Data risks: sample size, bias, timeliness, noise, inconsistent units, etc.
6. Impact of cleaning on subsequent modeling
7. If no data files: specify needed external data or parameter sources

## Output Requirements
- Use Markdown.
- Must be directly usable as material for `data_profile_report.md`.
- Do not write "suggest further analysis" -- provide specific recommendations.
- If field names conflict with the S1 symbol table, flag the inconsistency.
```

The sub-agent writes the result directly to `data/analyst_s2.md`.

---

### Step 3: Main Agent Produces Governance Results

The main agent synthesizes S1 and the temporary sub-agent output to produce:
- `data/data_profile_report.md`
- `data/data_dictionary.md`
- Cleaning results or description files under `data/processed/`
- `data/data_status.md`

Governance results must clearly indicate:
- Which fields are core variables
- Which fields are auxiliary variables
- Which fields are untrustworthy or unusable
- Whether data is sufficient for S3 candidate model screening

---

### Step 4: Fallback and Remediation

If data issues are severe, the main agent must record:
- Missing fields
- Unit conflicts
- Unfixable outlier items
- External sources that need supplementation

If necessary, document a "degraded modeling strategy" in `data/data_status.md`.


### Gate Failure -> Auto Rework (Hard-Coded, Cannot Be Skipped)

```bash
IF gate_check returns FAIL:
    1. Record failed gate name and failure reason (incomplete data availability conclusion / missing field-to-variable mapping / incomplete data profile / missing data dictionary / untraceable cleaning plan)
    2. Run python CUMCM_Workspace/_skill/scripts/stage_executor.py rework S2 --reason "Gate G1.5 failed: <specific reason>"
    3. Return to Step 2 (re-invoke Analyst sub-agent for data governance)
    4. rework count +1
    5. If rework >= 2: log to state/decision_log.json, generate failure report, block and prompt human intervention
    6. If failure reason is "upstream problem analysis incomplete": run python CUMCM_Workspace/_skill/scripts/stage_executor.py rework S1 --reason "S2 found incomplete problem analysis: <reason>", fallback to S1
```

**This logic cannot be omitted. Gate failure must execute the rework command, record count, and block on exceed.**
