# 建模阶段防错手册（按题型索引）

## 采用方式

建模启动前，依据 问题分析.md 判定各个子问题的题型，随后查阅相应章节的防错条目。

在 建模报告.md 末尾写一行：`本题涉及题型：[X, Y, Z]，已对照防错手册审查。`

⛔ 一个子问题可能涉及多个题型（如"用微分方程建模+优化控制参数"），务必与此同时查阅全部有关章节。

## 一、优化类（规划/调度/选址/路线/装箱）

### 务必做

- [ ] 清晰标明写出目标函数的优化方向（min 还是 max），不可含糊

- [ ] 列出全部约束条件的完整数学表达式（不可只在文字中提到）

- [ ] 各个决策变量标注类别（连续/整数/0-1）和取值界限（bounds）

- [ ] 多目标问题务必阐明权重来源（层次分析/归一化/题目给定），不可拍脑袋

- [ ] 大M法的M值务必标注取值依据（M > 变量可能的最大值的2倍以上）

- [ ] 校验可行域非空：不少于构造一个满足全部约束的可行解

- [ ] 非凸问题务必标注"需多起点求解"或"启发式算法"

### 禁止做

- [ ] 禁止遗漏非负约束——全部物理量（数量/距离/时间/成本）默认 ≥ 0，务必明示写出

- [ ] 禁止把整数变量当连续变量建模后不阐明松弛策略和取整方案

- [ ] 禁止约束只写一半（如只写上界不写下界，或只写等式不写不等式）

- [ ] 禁止贪心算法声称"最优解"——贪心仅可得到近似解，务必标注

- [ ] 禁止多目标权重全部相等且不给理由

### 交付界限预判

- 目标函数值应在什么量级？（成本几万？利润几百万？时间几小时？）

- 决策变量的合理界限？（不可出现负数车辆、小数人数、超容量库存）

- 约束是否互相矛盾？（预算约束+质量约束+时间约束能否与此同时满足？）

### 高频陷阱

- 约束互相矛盾导致可行域为空 → 建模时务必校验不少于出现一个可行解

- 非凸问题用凸优化方法 → 务必阐明凸性，非凸则标注"需多起点求解"

- 组合爆炸问题用枚举 → 变量超过20个时务必用启发式/近似算法

- 目标函数量纲不统一 → 多目标加权前务必无量纲化

### ⛔ 建模合理性与算法有效性约束（通用，适用于全部优化/建模工作项）

**问题本质：** 结果在物理约束内，但建模策略本身无效（黑箱暴力搜索高维空间、算法卡在局部最优、新增资源未被利用）。以下四条是结构性约束，不涉及具体题号或数值，长期适用。

**① 资源单调性约束**

当后续子问题相对前序子问题拥有严格更多的决策资源（更多决策变量/更多可用个体/更大的可行域涵盖关系）时，其最优目标函数值务必严格优于前序。

- 违反 → 判定为建模降维失败，务必回到建模层重新分解

- 禁止直接把"资源更多但结果更差/持平"的结果写入论文

- 建模报告中务必写出各个子问题的"资源递进关系"和"预期改善方向"

**② 解析基准下界约束**

任何启发式/元启发式优化（GA/DE/PSO/SA/贝叶斯等）产生的数值解，务必先有一个通过几何/物理/代数推导得到的解析可行解作为下界参考。

- 若数值解 ≤ 解析下界 → 判定为算法失败（非建模失败），务必核验编码/约束/罚函数

- 解析基准的构造方法务必在建模报告中写明（贪心分配/几何最优点/均匀分布/物理极限）

- 禁止"没有基准就直接跑优化随后接受结果"

**③ 搜索空间健康度约束**

在启动任何决策维度 ≥ 6 的黑箱优化前，务必先采样 N≥100 个随机点评估适应度分布。

- 若非平凡解（适应度 > 0 或优于朴素解）比例 < 30% → 判定搜索空间过稀疏

- 务必先做：变量降维 / 分层优化 / 启发式种群初始化

- 禁止在稀疏空间中纯随机初始化后直接跑优化

**④ 维度分层阈值**

当决策变量维度 ≥ 10 时，禁止单层黑箱优化。

- 务必明示完成变量分组/分层/序贯分解

- 建模报告中务必给出分层依据（资源独立性/时间尺度分离/几何解耦/因果链分割）

- 每层的维度应 ≤ 6，超过则继续分层

**⑤ 离散取整可行性约束**

连续松弛求解后对整数变量取整时，取整结果可能违反约束（超容量/超预算/不满足逻辑关系）。

- 取整后务必重新代入全部约束函数校验可行性

- 若取整后不可行 → 禁止直接采用不可行解，务必用修复启发式（贪心减少/邻域搜索/向下取整优先）

- 建模报告中务必写明取整策略（四舍五入/向下/向上/修复启发式）及其对目标函数的影响估计

- 禁止"取整后不校验就直接用"——这是导致终版方案不可行的最常见缘由

**⑤-b 松弛间隙（Relaxation Gap）监控**

连续松弛解与整数解之间的目标函数差（integrality gap）是衡量模型质量的关键指标。

- 务必与此同时报告松弛解和整数解的目标函数值

- 若 gap > 20% → 阐明整数约束对问题影响很大，取整策略务必更精细（不可简单四舍五入）

- 若 gap ≈ 0% → 阐明整数约束几乎不起作用，核验是否遗漏了关键的整数约束（如"人数务必为整数"）

- 建模报告中务必写明预期的 gap 界限，编码阶段对照校验

**⑥ 多目标归一化次序约束**

当目标函数涵盖不同量纲的多个分量时（如成本万元级 + 时间小时级 + 距离公里级），归一化次序直接决定最优解。

- 务必先对各个目标独立归一化到 [0,1]（极差法/Z-score），随后再加权求和

- 禁止不同量纲的目标直接加权（较多纲项会完全主导结果，小量纲项形同虚设）

- 归一化方法和权重来源务必在建模报告中写明

- 校验方法：去掉某个目标后，排名/最优解是否剧烈变化？若去掉某目标后结果几乎不变，阐明该目标被淹没了

**⑦ 时间/空间离散化步长约束**

离散化步长太大会跳过关键事件（碰撞瞬间、阈值穿越、峰值、运行状态切换），导致结果严重偏差。

- 步长务必 ≤ 系统最短特征时间的 1/10（如振荡周期0.1s → 步长≤0.01s）

- 空间网格务必 ≤ 最小特征尺度的 1/5（如物体宽度0.3m → 网格≤0.06m）

- 务必做步长收敛性校验：步长减半后结果变化 < 1% 才算收敛

- 禁止"拍脑袋选步长"——务必有物理依据或收敛性校验

- 事件检测型问题（碰撞/穿越/切换）务必用事件驱动而非固定步长

**⑧ 约束耦合度与可行域体积预判**

约束之间可能出现强耦合——独立看各个约束都合理，但组合后可行域极小甚至为空。

- 建模阶段务必分析约束之间的耦合关系：哪些约束会互相"挤压"可行域？

- 若 N 个约束中有 ≥3 个与此同时约束同一组变量 → 务必校验这些约束的交集非空且有充足体积

- 校验方法：随机采样 1000 个点，统计满足全部约束的比例（可行比例）

  - 可行比例 < 1% → 可行域极小，优化器很难定位可行解，务必放松某些约束或分层求解

  - 可行比例 < 0.01% → 几乎无可行解，建模有问题（约束互相矛盾或过紧）

- 建模报告中务必写明"约束耦合分析"：哪些约束是独立的，哪些是耦合的

- 若优化器返回的最优解恰好在多个约束的交点上（多个约束与此同时取等号）→ 高度怀疑可行域退化为低维流形，需核验是否遗漏了自由度

**⑨ 目标函数灵敏度方向预判（防止"死变量"）**

建模阶段务必预判目标函数对各个决策变量的灵敏度方向和量级。

- 对各个决策变量 x_i，务必回答：x_i 增大时，目标函数是变好还是变差？变化量级是多少？

- 若某个决策变量对目标函数几乎无影响（偏导数 ≈ 0）→ 阐明：

  - 该变量可能不应是决策变量（应是参数或常数）

  - 或目标函数遗漏了与该变量有关的项

  - 或该变量被其他约束完全锁死了（只有一个可行值）

- 建模报告中务必写出"灵敏度方向表"：

| 决策变量 | 增大时目标函数方向 | 预期灵敏度量级 | 约束限制 |

|----------|-------------------|---------------|---------|

| x_1 | ↓（减小=更优） | 高（主导项） | x_1 ∈ [0, 100] |

| x_2 | ↑（增大=更优） | 中 | x_2 ≤ budget/price |

| x_3 | ≈0（几乎无影响） | ⚠ 需核验 | — |

- 编码阶段校验：若真实灵敏度方向与预判相反 → 目标函数或约束写错了，务必回查

**⑩ 多目标 Pareto 前沿退化检测**

多目标优化中，若 Pareto 前沿退化（全部非支配解几乎相同），阐明模型有结构性问题。

- Pareto 前沿退化为一个点 → 目标之间不出现真正的冲突，多目标退化为单目标

  - 缘由1：约束太紧，可行域只有一个点

  - 缘由2：目标函数之间高度正有关（本质上是同一个目标的不同表达）

  - 缘由3：某个目标完全主导（权重/量级差异太大）

- Pareto 前沿退化为一条线（一维）→ 只有两个目标真正冲突，其他目标是冗余的

- 检测方法：计算 Pareto 前沿上各解的目标函数值标准差

  - 全部目标的标准差 < 目标值的 1% → 判定为退化

- 退化时务必：重新审视目标函数定义，核验是否有目标被淹没或约束过紧

**⑪ 优化结果稳定性校验（防止伪最优）**

启发式算法的结果可能是"运气好碰到的局部最优"而非真正的全局最优。

- 务必用不同随机种子跑 ≥5 次，统计目标函数值的分布：

  - 标准差 / 均值 < 1% → 结果稳定，可信

  - 标准差 / 均值 在 1%-5% → 结果基本稳定，取最优的那次

  - 标准差 / 均值 > 5% → 结果不稳定，阐明搜索不充分或问题有多个局部最优

    - 务必增加种群大小/迭代次数，或换更强的算法（如 CMA-ES/SHADE）

    - 或做变量降维后重新优化

- 建模报告中务必写明"稳定性校验方案"：跑几次、用什么种子、可接受的变异系数阈值

- 编码阶段务必实施稳定性校验并报告结果

**⑫ 约束活跃性分析（防止"虚假约束"和"遗漏约束"）**

求解完成后，务必核验各个约束的活跃运行状态（是否取等号/接近等号）。

- 全部约束都不活跃（解在可行域内部远离边界）→ 两种可能：

  - 约束太松（没有真正限制解空间）→ 核验约束参数是否无误

  - 遗漏了关键约束（真正的限制没写进模型）→ 回到题目重新复核

- 某个约束活跃但物理上不应活跃 → 模型有问题

  - 例：预算约束活跃（花光了全部预算）但题目暗示"预算充足" → 核验成本计算是否无误

- 多个约束与此同时活跃且解在它们的交点上 → 解可能是角点解（非光滑点）

  - 角点解对参数微扰极其敏感 → 务必做灵敏度分析

- 编码阶段务必交付约束活跃性报告：

```python

# 约束活跃性分析（优化求解后必做）

def analyze_constraint_activity(solution, constraints, tol=1e-4):

    """分析每个约束的活跃状态。"""

    report = []

    for name, (func, bound, direction) in constraints.items():

        value = func(solution)

        slack = abs(value - bound) if direction == 'eq' else (value - bound if direction == 'ge' else bound - value)

        status = 'ACTIVE' if abs(slack) < tol else f'slack={slack:.4f}'

        report.append(f"  {name}: value={value:.4f}, bound={bound}, status={status}")

    return '\n'.join(report)

```

**⑬ 时间窗/前驱约束的可行性预校验（VRP/调度类必做）**

时间窗约束和工序前驱约束是 VRP/调度问题中最常见的不可行来源。

- **时间窗可行性**：若客户 i 的时间窗为 [a_i, b_i]，从 depot 到 i 的最短旅行时间为 t_i，则务必满足 t_i ≤ b_i（反之无论如何都赶不到）

  - 建模阶段务必核验：是否出现任何客户的时间窗在物理上不可达？

  - 若出现 → 该客户务必被标记为"可选服务"或"准许迟到（带惩罚）"

- **前驱约束无环校验**：工序依赖关系务必形成 DAG（有向无环图）

  - 若出现环（A 依赖 B，B 依赖 C，C 依赖 A）→ 问题无解，建模有误

  - 编码阶段务必用拓扑排序校验无环性

- **时间窗传播**：若工作项 A 务必在工作项 B 此前完成，且 A 的最早完成时间 > B 的最晚启动时间 → 不可行

  - 务必做前向传播（计算最早启动时间）和后向传播（计算最晚启动时间）

  - 若某个工作项的最早启动时间 > 最晚启动时间 → 不可行，务必放松约束

**⑭ 对称性破缺（同质资源问题必做）**

当问题中有多个同质资源（同型号车辆/同规格机器/同能力人员）时，解空间出现较多对称解。

- N 个同质车辆的路线分配有 N! 种等价排列 → 搜索空间被放大 N! 倍

- 务必加入对称性破缺约束：

  - 车辆编号约束：车辆 1 的第一个客户编号 < 车辆 2 的第一个客户编号

  - 负载排序约束：车辆 1 的总负载 ≥ 车辆 2 的总负载

  - 字典序约束：车辆路线按字典序排列

- 不破缺对称性的后果：

  - 精确求解器（MIP）：分支定界树膨胀，求解时间指数增长

  - 启发式算法：较多时间浪费在等价解之间跳转，收敛极慢

- 建模报告中务必识别对称性并写明破缺策略

**⑮ 动态/多阶段问题的运行状态传递一致性**

多阶段决策问题中，前一阶段的决策结果是后一阶段的初始运行状态。

- 运行状态传递务必完整：前一阶段结束时的全部运行状态变量都务必传递到后一阶段

  - 常见遗漏：只传递了"位置"忘了传递"剩余容量/剩余时间/疲劳度"

- 阶段边界的连续性：

  - 位置连续：阶段 k 结束位置 = 阶段 k+1 启动位置

  - 资源连续：阶段 k 剩余资源 = 阶段 k+1 初始资源（除非有补给）

  - 时间连续：阶段 k 结束时间 + 转换时间 = 阶段 k+1 启动时间

- 校验方法：在阶段边界处核验全部运行状态变量的连续性

- 若用滚动时域优化（Rolling Horizon）：

  - 务必校验滚动窗口重叠区域的决策一致性

  - 窗口边界效应：靠近窗口末端的决策可能由于"看不到未来"而次优

**⑯ 不确定性处理与鲁棒性校验**

当问题参数有不确定性（需求波动/旅行时间随机/设备故障概率）时，确定性最优解可能在真实中不可行。

- 务必识别哪些参数有不确定性：

  - 需求量：一般情况下 ±20-30% 波动

  - 旅行时间：一般情况下 ±10-50% 波动（取决于交通状况）

  - 服务时间：一般情况下 ±20% 波动

  - 设备可用性：故障概率 1-10%

- 鲁棒性校验方法（不少于做一种）：

  - **场景分析**：在最好/最坏/最可能三种场景下评估解的表现

  - **蒙特卡洛模拟**：随机扰动参数 1000 次，统计解的可行率和目标函数分布

  - **鲁棒优化**：用最坏情况下的参数值求解（保守但安全）

- 若确定性最优解在参数扰动 ±10% 后有 > 20% 概率不可行 → 务必用鲁棒优化或加松弛

- 建模报告中务必写明"不确定性来源"和"鲁棒性校验方案"

**⑰ 子回路消除（TSP/VRP 类必做）**

TSP/VRP 的 MIP 建模中，若不加子回路消除约束，求解器会返回涵盖断开子回路的"最优解"。

- MTZ 约束（Miller-Tucker-Zemlin）：简单但松弛较弱，适合小规模

- DFJ 约束（Dantzig-Fulkerson-Johnson）：紧但数量指数级，需延迟产出（Lazy Constraints）

- 若用启发式算法：解的表示方式务必天然排除子回路（如排列编码）

- 校验方法：核验解中是否全部节点都在一条连通路线上

  - 若识别到子回路 → MIP 模型缺少子回路消除约束，务必补充

- 编码阶段务必在交付路线后校验连通性

## 二、微分方程/动力学/物理仿真类

### 务必做

- [ ] 全部运行状态变量标注物理含义、单位和取值界限

- [ ] 明示写出物理约束条件的数学表达式（接触约束/饱和/边界反弹）

- [ ] 标注系统是否刚性（stiff），推荐求解器（RK45 vs Radau/BDF）

- [ ] 初始条件务必有物理依据，标注来源（题目给定/稳态计算/物理推导）

- [ ] 若用数据驱动（如附件给的力/电流数据），务必核验数据的净偏差

- [ ] 守恒量（能量/质量/动量）务必列出，并标注"数值解中需监控"

- [ ] 开环积分类模型务必设计去漂移/反馈机制

### 禁止做

- [ ] 禁止"纯数学ODE求解"——务必在模型中写入物理约束后再求解

- [ ] 禁止忽略接触/碰撞约束（实体不可穿透、间隙不可为负、位移有上限）

- [ ] 禁止用固定步长欧拉法解高频/刚性系统

- [ ] 禁止不校验守恒量就交付结果

- [ ] 禁止把"数据是题目给的"当作结果超出物理极限的理由

### 交付界限预判

- 位移/速度/加速度的物理极限是多少？（题目是否给了最大间隙/最大速度？）

- 长时间积分后运行状态变量是否会漂移到无穷？（净力/净冲量是否为零？）

- 数值精度是否充足？（步长 vs 系统最高频率，不少于10倍采样）

### 高频陷阱

- 数据净冲量≠0 → 积分后运行状态变量单调漂移 → 务必设计去漂移机制

- 开环积分无反馈 → 误差累积 → 务必加阻尼/反馈/事件检测

- 守恒量（能量/质量）在数值解中漂移 → 务必监控并报告偏差

- 刚性系统用明示方法 → 需极小步长或直接发散

## 三、统计/回归/预测类

### 务必做

- [ ] 清晰标明训练集/测试集划分方式（时序问题务必按时间划分，不可随机）

- [ ] 标注各个变量的正负向含义（X增大时Y应增还是减）

- [ ] 多元回归务必核验共线性（VIF>10 的变量需处理）

- [ ] 预测模型务必给出置信区间/预测区间，不可只给点估计

- [ ] 模型选取务必有依据（为什么用线性/非线性/树模型/神经网络）

- [ ] 异常值处理策略务必阐明（剔除/缩尾/稳健回归）

### 禁止做

- [ ] 禁止用全部数据做归一化/特征工程后再划分（数据泄露）

- [ ] 禁止时间序列用随机 k-fold（务必用 TimeSeriesSplit 或滚动窗口）

- [ ] 禁止 R²<0.5 的模型直接采用（务必改进模型表现形式或阐明领域合理性）

- [ ] 禁止只报告显著变量不报告全部变量（p值挖掘）

- [ ] 禁止线性外推到训练数据界限之外而不加警告

- [ ] 禁止因果方向反推（X→Y 不可写成 Y→X）

### 交付界限预判

- 预测值应在什么界限？（历史数据的 min~max，不可超出太多）

- R² 预期多少？（物理/工程 >0.8，社科截面 0.2-0.6 可接受）

- 残差应是什么分布？（正态、零均值、等方差）

### 高频陷阱

- 线性模型拟合非线性数据 → 残差有弯曲模式 → 需加非线性项或换模型

- 外推预测发散 → 远期预测务必核验是否单调增长到不合理值

- 异常值拉偏回归 → 务必做稳健回归或剔除异常值后对比

- 过拟合 → 训练R²很高但测试R²很低 → 需正则化或减少特征

## 四、评价/决策类（AHP/TOPSIS/熵权法/DEA）

### 务必做

- [ ] 清晰标明标注各个指标的方向（正向=越大越好 / 负向=越小越好）

- [ ] 权重之和务必精确等于1（误差<0.001）

- [ ] AHP 务必计算并报告一致性比率 CR（CR<0.1 才可用）

- [ ] 终版排名务必做灵敏度分析（权重微调±10%后排名是否稳定）

- [ ] 无量纲化方法务必阐明（极差法/Z-score/比值法）并统一

### 禁止做

- [ ] 禁止正负向指标不做区分直接加权（会导致排名反转）

- [ ] 禁止全部权重相等且不阐明理由（等权法需有依据）

- [ ] 禁止评价结果无区分度（全部方案得分差<1%时务必换方法或调权重）

- [ ] 禁止混用不同量纲的指标直接加权（务必先无量纲化）

### 交付界限预判

- 综合得分应在 [0, 1] 界限内（TOPSIS）或 [0, ∞)（DEA）

- 排名应与常识保持一致（明显好的方案不应排最后）

- 权重分布应合理（不应某个指标权重>0.8 而其他<0.05）

### 高频陷阱

- 量纲不统一 → 混用百分比和绝对值 → 某个指标主导结果

- 指标间高度有关 → 信息重复 → 需降维或合并

- 排名与常识矛盾 → 核验是否正负向搞反

- 熵权法在样本量小时权重不稳定 → 需结合主观权重

## 五、图论/网络流/路线规划类

### 务必做

- [ ] 清晰标明图的类别（有向/无向、加权/无权、是否准许负权、是否有容量）

- [ ] 校验图的连通性（不连通时分别处理各连通分量）

- [ ] 网络流问题务必写出流量守恒方程（各个节点：流入=流出±源汇）

- [ ] 路线问题务必标注全部约束（容量/时间窗/车辆数/里程限制）

- [ ] 算法选取务必匹配图的特征（负权→Bellman-Ford，无负权→Dijkstra）

### 禁止做

- [ ] 禁止有负权边时采用 Dijkstra（务必用 Bellman-Ford 或 SPFA）

- [ ] 禁止忽略容量约束（边/节点的通过能力上限）

- [ ] 禁止把有向图当无向图建模（单行道/管道方向）

- [ ] 禁止 TSP/VRP 用贪心后声称"最优解"（仅可说"近似解"）

### 交付界限预判

- 最短路长度应 ≥ 0（无负权时）

- 最大流 ≤ 源点出边容量之和

- TSP 路线长度应在 [最小产出树, 2×最小产出树] 界限内（近似）

### 高频陷阱

- 最短路结果为负 → 有负权环 → 问题无解，务必报告

- 最大流实现时忘记加反向边 → 结果偏小

- 图不连通但算全局最短路 → 返回 inf 或错误路线

- 有向图建成无向图 → 路线不可行（如单行道）

## 六、几何/空间优化/布局类

### 务必做

- [ ] 全部实体用完整几何描述（长×宽×高/半径/外轮廓多边形顶点坐标）

- [ ] 碰撞检测务必用实体完整外轮廓，不可只用中心点距离

- [ ] 坐标系务必统一（全局坐标 vs 局部坐标，清晰标明标注原点和方向）

- [ ] 旋转角度统一单位（弧度 or 角度，全文保持一致，推荐弧度）

- [ ] 场地边界约束务必明示写出（物体任何部分不可超出边界）

- [ ] 对称性分析：是否可利用对称性减少搜索空间

### 禁止做

- [ ] 禁止未声明的降维简化（矩形→线段、体积→面积、实体→质心点）

- [ ] 禁止用中间计算坐标替代物理实体参数（用完整尺寸，不用中心距代替全长）

- [ ] 禁止忽略物体的真实尺寸只用参考点判定碰撞

- [ ] 禁止 AABB 碰撞检测用于旋转后的矩形（务必用 OBB 或 SAT）

### 交付界限预判

- 全部坐标应在场地边界内

- 物体间距离 ≥ 0（不可重叠/穿透）

- 布局面积利用率应合理（不可>100%，也不应<10%）

### 高频陷阱

- set_aspect('equal') 后坐标轴比例变化 → 视觉上的"不碰撞"可能真实碰撞

- 旋转后的矩形碰撞检测 → 不可用 AABB，务必用 OBB 或 SAT 分离轴定理

- 连续空间离散化 → 网格精度不够会漏掉可行解

- 忽略物体厚度/宽度 → 碰撞检测失效

### ⛔ 物理机理+优化复合题专项准则（几何/运动学/光学/流体等与优化结合的题型）

**适用场景：** 题目涉及物理/几何机理建模（微分方程/几何方程/光学方程）与参数优化的结合，需大规模实体的空间关系判定。

**① 多刚体/多实体链式约束传播**

当 N 个实体通过铰接/跟随/串联关系连接时，前一个实体的运行状态决定后一个的运行状态。

- 误差传播：第 k 个实体的位置误差 ≈ k × 单步误差（线性累积）或更差（非线性累积）

- 务必从链头到链尾正向传播计算，不可各自独立计算后拼接

- 链尾实体的约束（如速度上限/曲率限制）务必反向传播到链头的决策变量

- 校验方法：计算链尾实体的位置/速度，核验是否满足物理约束

- 若链长 N > 50，务必核验数值精度是否充足（float64 在 N>100 时可能有问题）

- 建模报告中务必写明"约束传播方向"和"误差累积估计"

**② 大规模几何判定的计算效率**

当 N 个实体需两两判定空间关系（遮挡/碰撞/覆盖）时，朴素 O(N²) 算法可能太慢。

- N < 100 → O(N²) 可接受，直接两两判定

- N 在 100-1000 → 务必用空间索引加速（KD-Tree/R-Tree/网格划分）

- N > 1000 → 务必用分层/分区策略，先粗筛再精判

- 加速结构不可改变判定结果的正确性——务必校验加速版和朴素版在小规模上结果保持一致

- 建模报告中务必写明"计算复杂度分析"和"加速策略"

- 编码阶段务必在小规模（N=10）上对比加速版和朴素版的结果

**③ 坐标系转换的精度累积**

多次坐标系转换（旋转+平移+投影）会累积浮点误差。

- 每一次旋转矩阵乘法引入 ≈ 1e-16 的相对误差

- 连续 K 次变换后误差 ≈ K × 1e-16（最好情况）到 K² × 1e-16（最差情况）

- 若涉及三角函数嵌套（如 arctan(sin(θ)/cos(φ))），精度损失更严重

- 务必做的校验：

  - 正变换后逆变换，核验是否回到原点（误差应 < 1e-10）

  - 旋转矩阵的正交性核验：R^T × R 应 ≈ I（误差 < 1e-12）

  - 若精度不够 → 用四元数代替欧拉角，或用符号计算简化表达式

- 建模报告中务必写明"坐标系定义"和"变换链"（从哪个系到哪个系，经过几步）

**④ 曲线上运动的曲率约束**

刚体在曲线上运动时，曲率半径限制了最大速度和最大角速度。

- 曲率半径 ρ = 1/κ，向心加速度 a = v²/ρ

- 若刚体有宽度 w，内侧曲率半径 = ρ - w/2，外侧 = ρ + w/2

- 当 ρ < w/2 时，内侧曲率为负 → 物理上不可能，阐明刚体无法通过该弯道

- 务必核验：曲线上每一点的曲率半径是否 > 刚体宽度的一半

- 速度约束：v ≤ sqrt(μ × g × ρ)（摩擦力给出向心力）或题目给定的速度上限

- 角速度约束：ω = v/ρ，若 ω 超过机械极限则速度务必降低

- 建模报告中务必写明"曲率分析"：曲线上曲率的最小值在哪里，相应的速度上限是多少

**⑤ 遮挡/阴影/视线判定的几何精确性**

光线遮挡判定务必考虑光源方向、实体几何、投影关系。

- 点光源（太阳近似）：遮挡判定 = 射线与实体的相交测试

- 面光源：遮挡判定 = 多条射线的统计（采样数 ≥ 100）

- 镜面反射：入射角 = 反射角，法线方向务必精确

- 常见错误：

  - 只判定中心点是否被遮挡，忽略边缘部分遮挡 → 务必判定面积遮挡比例

  - 用 2D 投影代替 3D 射线追踪 → 当实体有高度差时 2D 投影不精确

  - 忽略实体自身的厚度/高度 → 低矮实体可能被高实体完全遮挡

- 校验方法：选几个特殊角度（正午/日出/日落），手算遮挡结果，与代码对比

**⑥ 频域-时域一致性校验**

若与此同时用频域分析（FFT/传递函数）和时域仿真（ODE 数值积分），两者结果务必保持一致。

- 频域分析假设系统是线性时不变的（LTI）→ 若系统有非线性项，频域结果不精确

- 时域仿真可处理非线性，但计算量大

- 校验方法：对线性系统，频域和时域结果应保持一致（误差 < 1%）

  - 若不一致 → 核验是否有非线性项被频域分析忽略了

  - 若保持一致 → 可用频域结果加速计算（不需每一次都跑时域仿真）

- 建模报告中务必写明"线性假设的适用界限"和"非线性修正方案"

## 通用原则（全部题型适用）

### 单位与参数

1. **单位一致性**：符号阐明表各个量务必有单位列，全文单位统一（不可混用 m/cm/mm）

2. **参数可追溯**：各个数值参数标注来源（题目第X段/附件X/物理常数/文献[X]）

3. **参数集中定义**：全部物理参数在报告开头的符号阐明表中统一定义，不可散落在各处

### 约束与假设

4. **约束完整性**：题目中各个"不超过/不少于/务必满足"都要相应一个数学约束表达式

5. **假设必声明**：任何简化假设务必明示写出适用条件和误差估计

6. **隐式约束明示化**：物理接触（不可穿透）、非负（数量≥0）、守恒（进=出）等"显然"的约束也务必写成公式

### 子问题关系

7. **子问题递进**：后续问题务必引用前面问题的结果，不可各自独立

8. **数据传递**：问题N的输入参数若来自问题N-1的交付，务必清晰标明标注

9. **假设保持一致**：各子问题的假设不可互相矛盾

### 结果预判

10. **量级预估**：各个子问题的交付应在什么数量级？（提前估算，computational-realization 对照）

11. **方向预判**：加了约束/资源后，目标函数应变好还是变差？

12. **极端检验**：输入取极端值（0/最大/边界）时，模型交付是否合理？

### 实体表示（⛔ 最易犯错的通用准则）

13. **有体积的目标不可用点代替**：题目中描述的任何有尺寸的实体（圆柱体、矩形、车辆、建筑物），在判定条件中务必用其完整几何形状，不可简化为中心点/质心/参考点。

    - 遮蔽判定：务必遮住目标的完整轮廓（全部可见面），不是只遮住中心点

    - 碰撞判定：务必检测实体外轮廓是否重叠，不是只检测中心距

    - 覆盖判定：务必覆盖目标的完整面积/体积，不是只覆盖中心

    - 可见性判定：只要目标任何部分可见就算"未被遮蔽"，不是只看中心是否可见

14. **判定条件的严格性务必与题目保持一致**：若题目说"完全遮蔽"，就务必遮住100%；若说"有效遮蔽"，需清晰标明定义阈值（如遮住面积>90%）。不可自行降低判定标准。

15. **遮蔽/覆盖/碰撞判定的完整性（防止对象降维）**：

    - 判定务必作用于目标的**完整几何边界**（全部边界点或等价的充要条件）

    - 函数签名务必涵盖几何参数（半径/高度/长宽/外轮廓），**只传一个坐标点 = 反模式**

    - 禁止用"中心被遮蔽 = 整体被遮蔽"的等价假设，**除非数学证明两者等价**

    - 离散采样近似时务必做收敛性校验（如 N=100→300→500 结果稳定才算收敛）

    - 代码注释务必阐明：判定的是哪个几何体的哪些边界

    - **典型等价性证明（如有）**：若证明了"底面圆周被遮蔽⟹全部圆柱被遮蔽"，可只核验底面圆周；反之务必核验全部边界

16. **关键假设务必有数学证明或文献支撑（提升论文深度）**：

    - 任何涉及"等价性"、"简化合理性"的假设务必给出证明或反例分析

    - 不可只写"为简化问题，假设 XXX"，务必阐明为什么这个简化不影响结果

    - 典型例子：

      - 假设"时间区间连续" → 证明目标函数关于时间的连续性

      - 假设"离散采样充足" → 做收敛性校验（N=100/300/500 对比）

      - 假设"降维等价" → 证明低维判据 ⟺ 高维判据

    - 论文中的定理/引理/命题 ≥ 1 个（评审加分项）

17. **求解精度务必与方法匹配（避免粗糙求解）**：

    - 连续优化问题务必用高精度方法（二分查找/牛顿迭代/自适应步长），精度不少于 1e-6

    - 禁止用固定步长遍历（如 dt=0.01）作为终版结果——精度只到 2 位小数

    - 启发式算法（GA/DE/PSO）后务必用局部精化（如梯度下降/Nelder-Mead）

    - 结果交付位数务必与方法精度匹配（1e-6精度→6位小数；1e-2精度→2位小数）

18. **双算法交叉校验（关键结果必做）**：

    - 各个子问题的关键结果务必用**两种独立方法**互相校验

    - 方法组合举例：

      - 几何解析 + 启发式优化（差 <1% 阐明两者都对）

      - DE/GA + PSO/SA（差 <5% 阐明都收敛到同一最优）

      - 精确算法 + 近似算法（差可作为近似误差上界）

    - 单一算法结果 = 无法区分"真实最优"和"局部最优"

    - 建模报告中务必为各个子问题写明主算法+校验算法

19. **高维优化务必先做几何降维（避免盲目黑箱）**：

    - 维度 ≥ 10 的优化问题**禁止**直接喂给 DE/PSO/GA

    - 务必先做几何/物理分析降维：

      - 几何降维：利用对称性/视线关系/几何不变量

      - 物理降维：利用守恒律/量级分析/无量纲化

      - 变量分组：独立/半独立变量分层优化

    - 降维后每层维度应 ≤ 6

    - 建模报告务必写明降维的几何/物理依据，不可只写"分层优化"

20. **多资源协同优化务必用联合目标函数（防止资源闲置）**：

    - 多资源（多弹/多机/多点）协与此同时，目标函数务必是**联合覆盖**/**联合收益**，不可是各资源独立优化后叠加

    - 常见错误症状（检测到任何一个务必重做）：

      - 某个资源的关键指标 = 0（T=0、覆盖=0、贡献=0 等）→ 阐明该资源被误判为"无贡献"放弃了

      - 两个资源的作用区间**完全不重叠**（并集 = 加和）→ 阐明没有利用互补性

      - 某个决策变量取到了极端边界（如 t=44s 落在工作项结束后）→ 阐明对该资源的搜索空间或约束错了

    - 无误做法：

      - 目标函数评估时务必与此同时考虑全部资源，不可"单机独立最优化"随后放弃

      - 种子池务必涵盖"全部资源都有贡献"的启发式初始解（如几何推导的初值）

      - 约束务必保证各个资源被充分利用（如"各个弹 T_ind > 0"的硬约束，或惩罚项）

21. **启发式优化务必有几何/物理启动种子**（防止局部最优陷阱）：

    - DE/GA/PSO 等元启发式算法的初始种群**禁止**纯随机产出（高维下有效解比例极低）

    - 务必构造基于问题结构的启发式种子解（不少于 5-10 个）加入初始种群：

      - 几何启发：基于视线关系、拦截点预测、最近邻分配

      - 物理启发：基于守恒律、平衡态、极限工况

      - 贪心启发：基于局部最优/逐次选取

    - 种子解的质量务必校验（各个种子的目标函数值务必 > 随机解的中位数）

    - 若启发式解已经接近最优（差异 < 10%），阐明优化问题本质上不需黑箱搜索

22. **多阶段问题务必用分层优化而非全量优化**：

    - 当问题可分解为"工作项分配 + 各工作项独立优化"时，务必分两层求解：

      - 第一层（顶层）：资源分配/工作项划分（离散决策）

      - 第二层（底层）：各个子工作项独立优化（连续决策）

    - 禁止把全部维度（分配+优化）混在一起做全量搜索

    - 分层的依据务必在建模报告中写明（如"工作项间资源独立/时间独立/空间独立"）

    - 典型例子：多导弹多无人机问题 → 先分配"哪架无人机对哪枚导弹"，再各自优化每架无人机的航迹

23. **加权求和型目标函数务必做分量贡献度校验（防止目标被淹没）**：

    - 多目标加权求和后，务必分解报告各分量对总目标的贡献百分比

    - 若某个分量贡献 > 90% → 其他分量形同虚设，等价于单目标优化

      - 缘由1：归一化没做好（量纲差异导致大数吃小数）

      - 缘由2：权重设定不合理（某个权重远大于其他）

      - 缘由3：某个目标的数值界限远大于其他（即使归一化了，方差也可能差很多）

    - 若某个分量贡献 < 1% → 该目标被完全淹没，去掉它结果不变

      - 务必核验：这个目标是否真的不关键？若题目强调了它，阐明建模有问题

    - 校验方法：去掉某个目标后重新优化，若最优解几乎不变 → 该目标确实被淹没了

    - 建模报告中务必写明各目标的预期贡献比例界限

24. **优化结果的物理可解释性校验（防止"数学无误但物理荒谬"的解）**：

    - 优化器给出的最优解务必能用自然语言解释其物理含义

    - 以下情况务必标记为可疑并深入核验：

      - 调度问题：某台机器利用率 100% 而其他机器 0%（除非容量差异极大）

      - 选址问题：全部设施选在同一个位置（除非需求高度集中）

      - 路线问题：最优路线有明显的"绕路"（除非有时间窗约束）

      - 分配问题：某个客户/工作项被分配了 0 资源（除非约束清晰标明准许不服务）

      - 投资组合：全部资金投入单一资产（除非该资产 Sharpe 比率远超其他）

    - 可疑解的处理：

      - 核验约束是否遗漏（如"每台机器不少于分配 X 工作项"）

      - 核验目标函数是否遗漏了平衡项/公平性项

      - 若确实是最优解（物理上合理），在 计算结果.md 中解释为什么

    - 建模报告中务必写明"解的可解释性预期"：最优解应长什么样？

25. **收敛性判据务必量化（防止"跑够迭代次数就停"）**：

    - 启发式算法的终止条件不可只是"跑完 N 代"，务必有收敛判据：

      - 连续 K 代最优解改善 < ε → 判定收敛

      - K 和 ε 务必在建模报告中指定（如 K=50, ε=0.1%）

    - 若达到最大迭代次数但未收敛 → 务必报告"未收敛"，不可当作最优解

      - 未收敛的处理：增加迭代次数/换算法/做变量降维

    - 精确求解器（LP/MIP）的收敛判据：

      - LP：对偶间隙 < 1e-8

      - MIP：相对间隙 (UB-LB)/LB < 1%（或题目精度要求）

      - 若 MIP 求解器报告 gap > 5% → 解不够好，需加 cutting planes 或换求解器

    - 编码阶段务必在 JSON 中登记收敛运行状态：`"converged": true/false, "convergence_metric": X`

26. **目标函数上下界夹逼（定位解的质量）**：

    - 对各个优化子问题，务必构造目标函数的理论上界和下界：

      - **下界构造**（最小化问题）：松弛整数约束/去掉最紧约束/用凸包近似

      - **上界构造**（最小化问题）：任何可行解都是上界（贪心解/随机可行解）

    - 最优解的质量 = (你的解 - 下界) / (上界 - 下界)

      - 质量 < 5% → 接近最优，可接受

      - 质量 5%-20% → 还有改进空间，但可用

      - 质量 > 20% → 解质量差，务必改进算法

    - 建模报告中务必写明上下界的构造方法

    - 编码阶段务必在 JSON 中登记：`"lower_bound": X, "upper_bound": Y, "solution": Z, "gap_pct": (Z-X)/(Y-X)*100`

    - 若无法构造紧的界（如高度非线性问题），不少于用"最差可行解"作为上界、"无约束最优"作为下界

27. **对偶信息提取与利用（LP/MIP 必做）**：

    - 线性规划/整数规划求解后，务必提取对偶变量（影子价格/边际成本）：

      - 影子价格 > 0 的约束 = 瓶颈约束（放松它能改善目标函数）

      - 影子价格 = 0 的约束 = 非瓶颈（放松它没用）

    - 对偶信息的用途：

      - 灵敏度分析：哪个约束最值得放松？（影子价格最大的那个）

      - 资源定价：每单位额外资源值多少钱？

      - 问题递进校验：Q2 增加资源后，相应约束的影子价格应下降

    - 编码阶段务必交付对偶信息报告：

    ```python

    # LP/MIP 求解后提取对偶信息

    if hasattr(result, 'dual') or hasattr(prob, 'constraints'):

        print("=== 对偶信息（影子价格）===")

        for name, constraint in prob.constraints.items():

            shadow_price = constraint.pi  # PuLP

            print(f"  {name}: shadow_price={shadow_price:.4f}")

            if abs(shadow_price) > 0.01:

                print(f"    → 瓶颈约束！放松 1 单位可改善目标 {shadow_price:.4f}")

    ```

    - 建模报告中务必预判哪些约束是瓶颈（基于物理直觉），编码阶段对照校验

28. **解的邻域鲁棒性分析（防止"脆弱最优"）**：

    - 最优解附近是否有很多"几乎一样好"的解？

    - 校验方法：对最优解的各个决策变量做 ±5% 扰动，登记目标函数变化：

      - 全部扰动后目标函数变化 < 1% → 解鲁棒（好事，论文中可强调）

      - 某个变量扰动 5% 导致目标函数变化 > 20% → 解脆弱（务必在论文中讨论）

      - 某个变量扰动后约束被违反 → 解在可行域边界上（角点解，参考⑫）

    - 编码阶段务必交付邻域分析结果：

    ```python

    def neighborhood_robustness(solution, objective_func, constraints, perturbation=0.05):

        """分析最优解的邻域鲁棒性。"""

        base_obj = objective_func(solution)

        results = {}

        for i, (name, val) in enumerate(solution.items()):

            for direction in [1, -1]:

                perturbed = solution.copy()

                perturbed[name] = val * (1 + direction * perturbation)

                try:

                    new_obj = objective_func(perturbed)

                    feasible = all(c(perturbed) for c in constraints)

                    change_pct = abs(new_obj - base_obj) / abs(base_obj) * 100

                    results[f"{name}_{'+' if direction>0 else '-'}{perturbation*100:.0f}%"] = {

                        "obj_change_pct": change_pct,

                        "feasible": feasible

                    }

                except:

                    results[f"{name}_{'+' if direction>0 else '-'}{perturbation*100:.0f}%"] = {"error": True}

        return results

    ```

    - 鲁棒性分析结果写入 `图表/robustness_results.json`，供论文灵敏度分析章节采用

29. **数值精度与条件数监控（防止"计算无误但精度不够"）**：

    - 涉及矩阵运算（回归/线性方程组/特征值）时，务必核验条件数：

      - cond(A) < 100 → 数值稳定，结果可信

      - cond(A) 在 100-10000 → 需注意，结果可能有 2-4 位有效数字损失

      - cond(A) > 10000 → 数值不稳定，结果不可信，务必做正则化/预处理

    - 涉及大规模求和/累积运算时，核验是否有灾难性抵消：

      - 两个大数相减得到小数 → 相对误差被放大

      - 解决方法：Kahan 求和/重新排列计算次序/用对数空间

    - 编码阶段务必在涉及矩阵运算的代码中加入：

    ```python

    import numpy as np

    cond = np.linalg.cond(A)

    print(f"矩阵条件数: {cond:.2e}")

    if cond > 1e4:

        print(f"⚠ 条件数过大 ({cond:.2e})，数值结果可能不可靠！")

        print(f"  建议：正则化 / SVD 截断 / 预条件处理")

    ```

30. **子问题间结果一致性自动化诊断与修复（资源单调性违反时的处理过程）**：

    - 当检测到后续问题结果不优于前序问题（违反资源单调性）时，按以下诊断树定位缘由：

    ```

    Q(n) 结果 ≥ Q(n-1) 结果（最小化问题，应该更小）

    │

    ├─ 检查1：Q(n) 的可行域是否包含 Q(n-1) 的最优解？

    │   ├─ 否 → 约束写错了（Q(n) 的约束比 Q(n-1) 更紧，不是更松）

    │   └─ 是 → 继续检查2

    │

    ├─ 检查2：把 Q(n-1) 的最优解代入 Q(n) 的目标函数，值是多少？

    │   ├─ 比 Q(n) 的"最优解"更好 → Q(n) 的优化器没找到真正的最优（搜索不充分）

    │   └─ 比 Q(n) 的"最优解"更差 → 继续检查3

    │

    ├─ 检查3：Q(n) 新增的资源/变量是否真的被利用了？

    │   ├─ 新增变量取值 = 0 或取到边界 → 目标函数没有正确反映新资源的贡献

    │   └─ 新增变量有非零取值 → 继续检查4

    │

    └─ 检查4：Q(n) 的目标函数定义是否与 Q(n-1) 一致？

        ├─ 不一致（如 Q(n) 多了惩罚项）→ 不是真正的"更多资源"，是"更多约束"

        └─ 一致 → 建模有结构性问题，需要回到 建模报告.md 重新设计

    ```

    - 修复优先级：核验1（约束错误）> 核验2（搜索不充分）> 核验3（目标函数遗漏）> 核验4（问题定义不一致）

    - 编码阶段检测到违反后，务必按诊断树逐步排查，不可直接"接受结果"

    - 在 计算结果.md 中务必登记诊断过程和修复结果

### 建模交付标准（⛔ 最关键）

**关键原则：建模阶段不可只交付公式和方法名。编码阶段是纯实施者，遇到"结果不对"仅可按预案操作，严禁自行发明修正方法。**

建模报告务必在末尾涵盖以下 5 项，缺任何一项都不可结束当前环节：

15. **结果约束清单**：各个交付量的硬边界（超出即判定代码有误，不是"需讨论"）

16. **预期行为描述**：合理结果的定性特征（时间尺度/稳态/瞬态/单调性），不是具体数值

17. **异常处理预案**：每种可能的异常只给一种修正方法，不留选取空间。若建模阶段自己都不确定用哪种方法，阐明建模还没做完

18. **方法唯一性声明**：各个计算环节指定唯一方法（涵盖预处理/后处理/插值/滤波），禁止替代方案。这些"工程细节"恰恰是导致不同实现得到不同结果的根源

19. **验证检查点**：编码阶段务必实施的 pass/fail checklist，每项关联到异常预案

### ⛔ 方法唯一性的具体要求（第18条展开）

**任何涉及"选取"的环节都务必在建模阶段定死，编码阶段严禁自行决策：**

| 类别 | 务必指定的内容 | 不指定的后果 |

|------|--------------|-------------|

| 数据预处理 | 处理界限（从哪到哪）、方法名、参数值、锚点 | 不同实现得到不同结果 |

| 滤波/去噪 | 滤波类别、截止频率、窗口大小、阶数 | 保留的信号特征不同 |

| 插值/重采样 | 插值方法（线性/样条/最近邻）、目标采样率 | 峰值位置和幅度不同 |

| 去趋势/去漂移 | 起始时刻、拟合方法、多项式阶数、基准点 | 结果可能差10%以上 |

| 异常值处理 | 判定标准（几倍标准差）、处理方式（剔除/缩尾/替换） | 回归系数方向可能反转 |

| 特征工程 | 具体的变换公式、分箱边界、编码方式 | 模型输入不一致 |

| 求解器参数 | 精度(rtol/atol)、最大迭代次数、收敛判据 | 精度和收敛性不同 |

| 随机算法 | 随机种子、种群大小、迭代次数、交叉/变异率 | 每一次运行结果不同 |

**⛔ 准则：若建模报告没有精确指定某个环节的方法和参数，编码阶段严禁自行选取。务必视为建模未完成，在 计算结果.md 中标注"建模报告未指定XXX方法，需补充"。**

### ⛔ 方法选取的自校验标准（防止选错方法）

仅仅"指定唯一方法"还不够——若指定的方法本身就是错的呢？建模阶段务必与此同时给出**校验标准**，让编码阶段能判定方法是否选对了：

**对各个涉及"选取"的环节，建模报告务必写出：**

1. **选取依据**：为什么选这个方法而不是其他方法（物理缘由，不是"方便"）

2. **正确性判据**：用什么标准判定这个方法选对了（不是只看结果在约束内）

3. **对比校验**：若不确定，要求编码阶段与此同时实现两种方法并对比

**正确性判据的写法（建模报告中）：**

```markdown

## 方法验证标准

去漂移方法：从 t=0 开始线性去趋势

正确性判据：
```
