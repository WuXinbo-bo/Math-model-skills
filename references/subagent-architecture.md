# 角色协作架构

Meta-model-agent 采用“主控—实施—审计”三层闭环。角色名称表示职责边界，不绑定任何外部产品或运行平台。

## 1. 主控角色

- 载入 `任务包.json`、`执行协议.md` 与 `工作指南.md`；
- 判定阶段能否启动、是否返工与何时转入核验点；
- 维护 `状态/工作流状态.json` 与 `状态/事件日志.jsonl`；
- 统一调用 `validate`、`gate_check`、`complete`、`checkpoint` 和 `rework`；
- 负责跨阶段依赖与恢复次序。

## 2. 实施角色

- 仅处理当前阶段划定的工作项；
- 只查阅阶段包和工作区内的有效证据；
- 交付物务必可被独立复核；
- 发现上游矛盾时向主控报告，不擅自调整阶段次序。

## 3. 审计角色

- 不替代实施角色生产重点成果；
- 专注识别缺项、逻辑断层、数据陈旧、模板损坏和门禁风险；
- 结论务必引用磁盘文件、运行状态登记或门禁报告，不接受无证据判定。

## 4. 标准闭环

1. 主控角色发布阶段包；
2. 实施角色完成阶段内工作项；
3. 审计角色逐条核对门禁；
4. 缺陷在当前阶段内返工；
5. 先通过 `validate`，再运行 `gate_check`；
6. 出现核验点时等待 `approve / feedback / rerun`；
7. 证据闭合后才推进下一阶段。

## 5. 证据位阶

优先采用当前工作区文件、`审查/门禁/{工作名称}.json`、`状态/工作流状态.json` 和 `状态/事件日志.jsonl`。工作协议与运行参考实现用于解释契约，严禁替代真实完成证据。

## 6. 最小交接包

每一次 `begin` 后，`当前任务/执行中/` 至少包含：

- `任务包.json`
- `执行协议.md`
- `工作指南.md`
- `协作角色.json`
- `协作架构.md`
- `复核闭环.md`

基准模式维持完整契约；增强模式仅可提高审计、恢复与证据强度。

## 7. 冠军审稿职责隔离

冠军模式下，每轮不少于区分三个职责：独立审稿、论文调整和修复复核。审稿角色仅可写审稿报告，调整角色仅可依据修订计划产出完整新稿，复核角色务必重新查阅新稿校验，严禁由调整者自行确认通过。前三轮审稿视角务必相互独立。
