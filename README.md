面向数学建模研究与竞赛论文产出的工程化辅助项目。

Meta-model-agent 优先适配 Codex，同时兼容能够读取 Markdown Skill、执行本地命令并访问工作区文件的其他 AI 工具。项目围绕真实研究活动组织工作：读取题面与数据、定义问题、设计模型、运行实验、核验结果、构造创新、整理图表、撰写论文并实施多轮审稿，最终形成可追溯的研究成果。

> 本项目仅提供数学建模研究辅助，不代替使用者进行学术判断、事实核验、竞赛决策或提交确认。数据、模型、程序、引用、计算结果、论文内容及最终提交材料均应由实际使用者独立检查；使用项目产生的任何成果及其相关风险，由实际使用者承担。

## 项目定位

Meta-model-agent 致力于帮助研究者和参赛团队形成高质量、可复现、可验证的数学建模论文。它并不把“生成一篇文字完整的论文”视为唯一目标，而是强调研究全过程的证据一致性：

- 题意理解应来源于真实赛题和附件；
- 模型假设、变量、公式与约束应能够解释；
- 程序应真实运行，结果应能够复现；
- 图表和论文数值应与程序结果保持一致；
- 结论应有模型、实验或数据证据支撑；
- 最终材料应由使用者审核后再决定是否提交。

项目适用于 CUMCM、51MCM、MCM/ICM 以及结构相近的数学建模研究任务。不同竞赛的规则、格式和学术要求可能发生变化，使用前应以主办方当期正式文件为准。

## 具体研究工作流

### 读取题面并建立可信问题底稿

系统优先读取赛题原文、附件提取文本和用户提供的数据，不以模型记忆代替真实题面。随后逐句识别任务目标、已知条件、显式约束、隐含条件、模糊表述和交付要求，并形成：

- 子问题之间的依赖关系；
- 变量、参数、单位和符号表；
- 可直接使用的事实与仍需验证的假设；
- 数据缺口、附件用途和外部资料需求；
- 初步模型路线、实验计划和论文图表计划。

这一过程产出 `问题分析.md`。后续模型和论文必须以该文件与真实题面为依据，避免在执行过程中逐渐偏离原题。

### 把自然语言问题转化为数学机制

在问题结构明确后，系统为每个子问题确定输入、决策变量、状态变量、目标函数、约束、输出指标和验证方式。模型设计不是简单列举算法，而是回答：

- 研究对象如何表示；
- 变量之间存在什么机制关系；
- 哪些条件属于硬约束，哪些属于可调偏好；
- 参数如何估计或给定；
- 模型在什么条件下成立，又会在什么条件下失效；
- 子问题之间如何传递参数、状态和结果。

系统会把推荐方法与备选方法放在同一问题契约下比较，并将最终采用、放弃或组合某种方法的理由写入 `建模报告.md`。同时为程序实现导出可检查的公式、约束与数值范围。

### 建立简单基线，再讨论复杂方法

对于预测、优化、评价、分类、仿真或决策问题，先建立一个能够运行和解释的简单基线。例如使用朴素预测、线性模型、贪心可行解、等权评价或简化物理情形，得到最低可接受参照。

复杂模型只有在下列条件至少满足一项时才值得引入：

- 能解决基线无法表达的关键约束；
- 能显著改善预测、优化或评价指标；
- 能提高稳定性、鲁棒性或可解释性；
- 能把多个子问题统一到同一数学结构；
- 能处理题目中真实存在的非线性、动态性、不确定性或空间关系。

若复杂方法没有相对基线产生可验证收益，系统应回退到更简单、解释更清晰的方案，而不是为了看起来高级而保留复杂模型。

### 编写程序并开展真实计算实验

模型确定后，将每个子问题映射到独立程序、结果文件和验证逻辑。程序需要真实执行，不能用手工编造的结果代替运行输出。计算工作包括：

- 数据清洗、参数加载和随机种子控制；
- 基线与候选模型求解；
- 约束可行性和边界条件检查；
- 训练、测试或时间切分；
- 灵敏度、稳健性、误差和极端情景分析；
- 运行时间、收敛过程和关键中间量记录；
- 每个子问题结果的 JSON 化保存。

最终由 `程序/主程序.py` 汇总计算任务，并将主要数值写入 `计算结果.md` 与 `图表/全部结果.json`，供图表和论文直接读取。

### 将计算结果转化为论文证据

图表不是装饰，而是对模型结论的可视化证明。系统依据问题分析中的图表计划和真实结果文件，生成：

- 数据分布、趋势和相关关系图；
- 模型拟合、残差和误差诊断图；
- 优化过程、约束活跃性和方案对比图；
- 灵敏度、稳健性和情景分析图；
- 技术路线、模型关系和子问题流程图；
- 论文所需的汇总表、参数表和结果表。

每张图都应能够追溯到数据或程序输出，并在 `图表/图表引用.tex` 中形成稳定引用。正文必须解释图表呈现了什么、为什么出现这种结果，以及它对问题结论意味着什么。

### 组织完整论文，而不是拼接模型说明

写作过程以题目逻辑和证据关系组织章节，将问题分析、假设、模型、算法、实验、图表、结果解释、优缺点和结论连成完整叙事。论文中的关键数字必须与结果文件一致，摘要中的结论必须能在正文定位到对应模型、图表或实验。

系统重点检查：

- 摘要是否包含问题、方法、关键结果和结论；
- 各子问题是否都有建模、求解、验证和回答；
- 公式符号、程序变量和论文表述是否一致；
- 图表是否在正文中被引用和解释；
- 参考文献是否真实、格式统一且引用位置合理；
- 局限性和负面结果是否被诚实说明；
- 附录是否提供足以理解和复现工作的程序材料。

### 通过审稿反馈修改论文

冠军模式会从数学严谨性、证据与复现性、评委阅读体验等不同视角独立审稿。每轮都必须先给出带位置的问题清单，再形成修订计划，随后生成完整新稿，并由新的复核视角检查每一项修改是否真实生效。

若审稿发现公式、假设、算法或核心结果错误，工作流会回退到问题分析、模型或程序环节重新处理，而不是仅用文字弱化错误。只有 P0 问题清零、主要 P1 问题得到控制且综合评分达到内部质量门槛，修订稿才可进入编译与提交验收。

### 编译并检查最终材料

最后对 LaTeX 工程、PDF 和编译日志执行检查，包括匿名信息、页数、模板、引用、公式编号、图表嵌入、文件新鲜度和未解析错误。编译通过只代表工程满足已编码的检查合同，不代表模型与结论天然正确，最终仍需使用者人工通读和确认。

## 创新如何产生

Meta-model-agent 不把“使用冷门算法”或“叠加多个模型名称”直接视为创新。创新必须来源于题目中的真实矛盾，并通过计算或理论证据证明其必要性和有效性。

### 从问题结构中寻找创新点

系统首先寻找基线模型无法同时处理的部分，例如：

- 多个目标相互冲突；
- 约束会随时间或状态变化；
- 参数存在区间、概率或场景不确定性；
- 子问题共享变量，却被传统方法割裂处理；
- 数据有限，但存在明确物理或业务机理；
- 静态最优方案在动态环境中失效；
- 平均性能良好，但极端情况下风险过高。

这些矛盾决定创新方向。若题目不存在相应需求，就不应强行加入复杂机制。

### 形成多条候选创新路线

针对已识别的矛盾，系统可提出不同层次的改进：

1. **表示创新**：重新定义状态、网络、空间关系或评价指标，使问题结构表达得更准确。
2. **机制创新**：加入动态演化、反馈、竞争协同、风险传播或物理约束。
3. **目标创新**：把单一目标改造成多目标、分层目标、风险调整目标或公平性目标。
4. **约束创新**：引入鲁棒约束、机会约束、软约束惩罚或情景约束。
5. **模型融合**：将机理模型与数据模型结合，但必须说明两者各自解决什么问题。
6. **算法创新**：针对问题结构设计编码、邻域、启发式规则、分解策略或加速方法。
7. **验证创新**：设计反事实情景、压力测试、消融实验或结构性验证，使结论更可信。
8. **统一框架创新**：让多个子问题共享同一状态变量、目标体系或决策内核。

### 用对照实验筛选创新

每个候选创新都应与基线比较，而不是只展示自己的结果。比较内容包括：

- 目标值、预测误差或综合评价指标；
- 可行解比例和约束满足程度；
- 对参数扰动与数据噪声的敏感程度；
- 极端情景下的性能退化；
- 计算时间、收敛速度和实现成本；
- 可解释性和论文表达难度。

只有改进幅度、稳定性或解释能力具有实际意义，创新路线才进入最终模型。若收益很小、依赖偶然参数或大幅增加复杂度，应在报告中降级为探索性方案或直接舍弃。

### 用消融和反例证明创新有效

对于组合模型，应分别移除新增模块并重新计算，观察性能和约束是否明显退化；对于新目标或新约束，应比较加入前后的方案结构；对于动态或鲁棒机制，应构造静态、平均和极端情景进行对照。

创新结论至少需要回答：

- 新增机制解决了哪个原有缺陷；
- 没有该机制时会出现什么问题；
- 改进是否在多个参数或场景下仍然存在；
- 是否引入新的局限和计算代价；
- 论文中的创新表述是否与实验支持强度匹配。

### 在审稿中继续修正创新表述

审稿角色会专门攻击创新部分：检查是否只是换名称、是否缺少基线、是否存在数据泄漏、是否把微小提升写成重大突破、是否回避失败场景。修改论文时必须根据证据调整创新强度，必要时补充实验，或把未经充分证明的内容改写为局限与后续方向。

最终保留的创新应当是问题驱动、模型上可解释、程序上可实现、实验上可验证、论文中可清晰说明的改进。

## 质量模式

### Baseline

默认运行方式。保持研究活动的依赖顺序、交付物、检查点和门禁合同稳定，用于建立能够重复执行的研究基线。

### Enhancement

在基线完成后针对薄弱环节开展受控返工，可加强模型解释、程序验证、证据链、图表或论文质量，但不会绕过上游证据重新生成结论。

### Championship

面向更高质量目标。在 `MANUSCRIPT` 和 `ASSURANCE` 之间强制执行不少于三轮模拟审稿：

1. 数学与模型严谨性审稿；
2. 证据链与可复现性审稿；
3. 竞赛评委与表达质量审稿。

每轮均包含审稿报告、修订计划、完整修订稿和逐项复核。最终需满足综合分不低于 85、P0 问题为 0、P1 问题不超过 2，才可进入最终验收。

评分和门禁属于项目内部质量控制手段，不代表任何竞赛主办方、评委或学术机构的正式评价，也不能保证获奖、录用或取得特定成绩。

## 优先适配 Codex

本项目优先按照 Codex 的 Skill 与本地工作区协作方式设计：

- 使用根目录 `SKILL.md` 作为能力入口；
- 通过 `references/` 按需载入阶段协议和知识库；
- 通过 `scripts/` 管理工作区、状态、门禁和审稿轮次；
- 允许 Codex 在共享工作区中读取证据、执行程序并更新成果文件；
- 适合在 Codex 中持续执行、断点恢复和验证多文件任务。

在 Codex 中，可以使用类似提示开始：

```text
请使用 Meta-model-agent 辅助当前数学建模研究。
先读取赛题与附件，建立工作区，并按问题分析、模型设计、计算实验、证据整理、论文写作和质量审稿的依赖关系推进。
所有程序需要真实运行，论文中的关键数值必须能够追溯到结果文件。
在最终提交前启用冠军模式完成多轮审稿和修订。
```

具体的 Skill 安装位置和加载方式可能随 Codex 版本或运行环境变化，应以当前 Codex 提供的 Skill 配置方式为准。

## 兼容其他 AI 工具

Meta-model-agent 不依赖单一模型服务。其他 AI 工具若具备以下能力，也可以读取并执行本项目：

- 能够加载 Markdown 指令文件；
- 能够读取和修改本地工作区文件；
- 能够运行 Python 与必要的编译命令；
- 能够维护多轮任务状态；
- 能够按协议调用独立工作角色，或以顺序方式完成角色任务。

不同 AI 工具在上下文长度、指令遵循、文件编辑、子任务协作和命令执行方面存在差异，因此实际行为和产出质量可能不同。迁移到其他工具时，应首先检查路径、命令、权限、编码和子任务机制是否兼容。

## 快速开始

### 1. 初始化工作区

在需要处理的数学建模项目目录中运行：

```bash
python scripts/workspace_init.py --workspace . --competition cumcm --output-format pdf
```

竞赛模板必须在初始化时选择：

```bash
python scripts/workspace_init.py --workspace . --competition cumcm
python scripts/workspace_init.py --workspace . --competition 51mcm
python scripts/workspace_init.py --workspace . --competition mcm-icm
```

生成可编辑 Word 时使用：

```bash
python scripts/workspace_init.py --workspace . --competition cumcm --output-format docx
python 工具/docx_export.py --workspace .
```

选择结果会写入工作流状态，并实际控制论文模板、语言、匿名字段、字号与页数门禁。

门禁同时接受中文工作区的 `论文/章节/` 和模板原生的 `论文/sections/`，并读取实际 `.cls` 校验基础字号和关键版式合同；仅修改文档类名称不能绕过竞赛检查。

三套模板都包含完整章节工程，而非单个空白入口：

| Profile | 文档类与版式 | 完整章节 | 专用终验 |
| --- | --- | --- | --- |
| `cumcm` | `cumcmthesis`、中文字体、国赛电子稿 | 重述、分析、假设、符号、逐问模型、敏感性、评价、代码附录 | 中文摘要/关键词、匿名、模板和引用 |
| `51mcm` | `51mcmthesis`、内置宋体/楷体、`withoutpreface` | 重述、逐问求解、独立验证、稳健性、结论、复现及补充结果 | 报名号/题号/组别、禁止学校与成员联系方式 |
| `mcm-icm` | 自带 `mcmicm.cls`、Letter、12pt、Control Number 页眉 | Summary Sheet、数据、基线/主模型、结果、验证、建议、复现、AI 报告 | 英文、A--F 题号、匿名、25 页 solution 标签与 AI 披露 |

### 2. 查看当前工作状态

```bash
python scripts/pipeline_manager.py overview --workspace .
python scripts/stage_executor.py current --workspace .
```

### 3. 启动问题情境解构

```bash
python scripts/stage_executor.py begin DISCOVERY --workspace .
```

完成阶段工作后依次运行：

```bash
python scripts/stage_executor.py validate DISCOVERY --workspace .
python scripts/stage_executor.py gate_check DISCOVERY --workspace .
python scripts/stage_executor.py complete DISCOVERY --workspace . --artifacts "问题分析.md"
python scripts/stage_executor.py checkpoint DISCOVERY --workspace . --action approve --note "reviewed"
```

### 4. 启用冠军模式

```bash
python scripts/pipeline_manager.py set-mode championship --workspace .
```

完成 `MANUSCRIPT` 后启动多轮审稿：

```bash
python scripts/championship_review.py init --workspace .
python scripts/championship_review.py start-round --workspace . --round 1 --perspective mathematical
```

至少完成三轮并通过冠军门槛后执行：

```bash
python scripts/championship_review.py finalize --workspace .
```

## 常见问题

### 能否全自动完成数学建模论文？

项目能够自动化部分流程，但不建议将其视为无人监督的论文生成器。模型选择、结果解释、引用核验和最终提交都需要实际使用者参与。

### 是否保证获奖或达到冠军质量？

不保证。`championship` 是内部质量模式名称，只表示采用更严格的审稿和返工标准，不代表竞赛结果承诺。

### 是否只能在 Codex 中使用？

不是。项目优先适配 Codex，但其他能够读取 Skill、编辑文件并运行命令的 AI 工具也可尝试使用。兼容程度取决于具体工具能力。

### 生成的论文能否直接提交？

不建议未经人工检查直接提交。使用者应核验论文内容、数据、模型、程序、引用、匿名信息和竞赛规则，并对最终材料负责。

### 出现错误或中断怎么办？

先查看：

```bash
python scripts/stage_executor.py status --workspace .
python scripts/pipeline_manager.py next --workspace .
```

若上游成果存在问题，使用 `rework` 从对应阶段重新执行，避免仅修改最终论文掩盖模型或数据缺陷。
